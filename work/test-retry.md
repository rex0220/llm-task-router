# 記事テーマ: TypeScript/Node.js で堅牢なリトライ処理を実装する

外部API呼び出しのリトライを、指数バックオフ＋ジッターで安全に実装する方法を、動くコード例つきで解説する記事を作成してください。

## 対象読者

- 外部APIやマイクロサービスを叩く Node.js / TypeScript のエンジニア
- 「とりあえず3回リトライ」で済ませているが、もう少し正しくやりたい人

## 記事のゴール

- 単純な即時リトライがなぜ危険か（雪崩・二重課金・無駄負荷）を説明できる
- 指数バックオフとジッターの考え方と実装を理解する
- どのエラーを再試行すべきか／すべきでないかを判断できる

## 含めたい論点

- リトライ対象の選別：5xx / 429 / ネットワーク断は再試行、4xx（認証・バリデーション）は再試行しない
- 指数バックオフの式（base * 2^attempt）と上限（cap）
- ジッター（full jitter）でリトライの同期集中を避ける
- リトライ回数上限・全体タイムアウト・AbortSignal による中断
- べき等性（POSTの再試行リスク、Idempotency-Key）
- 実装例：fetch をラップした `retry()` 関数（依存最小、TypeScript）

## 制約・トーン

- コード例は実際に動く TypeScript にする（import・型・標準APIの実在を守る）
- 外部ライブラリは最小限にし、使う場合は理由を述べる
- 「必ず」「絶対」などの過度な断定を避け、トレードオフを示す
- Qiitaにそのまま貼れる読みやすい長さにまとめる
