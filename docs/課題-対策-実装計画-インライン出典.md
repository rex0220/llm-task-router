実データ確認（`llm-router-e2e/runs/2026-06-22-hayabusa2`）: `claims.json` は present/verified claim に `location.heading` と `sourceIds` を持つ。`sources.json` は `cited` 焼き込み済み。＝「どの見出しの主張が、どの検証済み source を参照するか」は**既に台帳にある**。足りないのは本文への出力だけ。

> **レビュー反映（2026-06-22）**: 当初版は `location.anchorHash` を「見出しアンカー」と誤解していた。実装は [claimsNormalize.ts:17](../src/cli/claimsNormalize.ts#L17)「identity は claim 文の hash のみ（**anchorHash と id の hash8 は同一値**）」のとおり、`anchorHash` は **claim 文のハッシュ（＝claim id の hash8）＝claim 単位**であり、見出しの識別子ではない。したがって本計画は **グループ化キー＝`location.heading`（テキスト）**、**ブロック ID と本文の節特定＝heading テキスト由来**で再設計する（#1 参照）。台帳に「見出しの安定 ID」は存在しない。

## 設計の確定事項

### 何を作るか（スコープの芯）

- **見出し（section）単位**で、その節の `present && verified` claim が参照する source への**出典マーカーを本文へ機械挿入**する。
- マーカーは**機械が生成・機械が検証**する。**LLM／人は書かない**（参考リンク計画と同じ捏造防止の原則。番号・URL を LLM に作らせない）。
- **profile 単位で opt-in**（既定 off）。技術・リファレンス系は on で信頼性を上げ、「物語として読み解く」一般向け読み物（例: hayabusa2, qiita 既定）はノイズになるため off。

### なぜ「文単位 inline [S001]」を採らないか（重要な制約）

[ClaimsSchema.ts:78](../src/schemas/ClaimsSchema.ts#L78) の通り、claim の位置情報は **`location: { heading, anchorHash }`**。このうち本文位置に使えるのは **`heading`（見出しテキスト）だけ**で、文・文字オフセットは台帳に無い（`anchorHash` は claim 文のハッシュ＝位置ではない。前掲レビュー反映）。さらに `claim` テキストは normalize 後の**言い換え**で、本文の逐語一致ではない。

したがって「特定の文末に [S001] を打つ」には、言い換え claim を本文の特定文へ fuzzy マッチする新工程が必要になり、

- revise のたびに再アンカーが要る（CLAUDE.md が最も嫌う **stale 台帳**の温床）
- 誤マッチ（別の文に出典が付く）の事故リスク
- 検証（verify-artifacts）がマッチ品質に依存して脆くなる

リターン（精度）に対しコストとリスクが見合わない。**台帳が持つ粒度（見出し）に出力を合わせる**のが本計画の核心判断。文単位 inline は**スコープ外**（将来、claim に本文オフセット/quote を持たせる別計画が前提）。

### マーカー方式（参考章と同じ「再生成で人の編集を壊さない」原則）

各対象節の**末尾**（次の見出し直前／EOF）に、**節キー（＝見出しテキストから決定的に導出した slug/hash）**で名前付けしたマーカーブロックを置く:

```md
## 持ち帰った砂が語ったこと：リュウグウ試料の科学的成果

…本文…

<!-- cite:begin:<headingKey> -->
> 出典: [S006](https://www.jaxa.jp/press/2022/06/20220610-1_j.html) / [S007](https://www.nature.com/articles/s41550-021-01550-6)
<!-- cite:end:<headingKey> -->
```

- ブロック ID（`headingKey`）は **inlineCite が見出しテキストから決定的に計算する**（`location.anchorHash` は claim 文ハッシュなので**使わない**。#1）。本文側の見出しからも同じ関数で再計算して突き合わせる。これで「**見出しが変わればキーが変わり旧ブロックは stale 化→除去**」という追従が成立する。
- ブロック内は決定的生成。**マーカー外**に編集長が書いた文・注記は保持。
- 既存 [src/cli/references.ts](../src/cli/references.ts) の `replaceMarkedBlock` は**単一ブロック前提**（`bCount===1 && eCount===1` を要求）。cite は**節数ぶんの多ブロック**なので、流用ではなく**多ブロック版を新規実装**する（挿入・更新・stale 除去・破損検査をすべて複数ブロックで。#4）。
- **節特定は見出しテキスト一致**で行う（台帳に見出しの安定 ID が無いため）。**同名見出しが複数ある場合は配置先が一意に決まらない**ので、「最初の一致に置く＋2件目以降は warning でスキップ」を既定とする（#5）。節末判定の見出しレベルは references の `^#{1,2}` に倣う（`###` 配下 claim も対象にするなら `^#{1,3}` へ拡張）。

### レンダリングモード（profile で選択）

- `marker`（既定）: 上記の節末「出典:」行（リンク列）。実装が単純で Qiita/Zenn/汎用すべてで動く。
- `footnote`: Qiita/Zenn の脚注記法に寄せる。節末に `[^cite-7a3f9c12]` を置き、記事末に `[^cite-7a3f9c12]: [S006](url) / [S007](url)` を機械生成（脚注定義も marker ブロックで管理）。読み口は軽いが媒体依存。**初期実装は `marker` のみ**、`footnote` は将来拡張（profile の `inlineCite.mode` で切替余地だけ用意）。

### 生成タイミング・バックアップ

- **claims-normalize と references の後**（`claims.json`/`sources.json` 確定後、参考章が入った final.md に対して）。
- 専用 CLI で `final.md` を機械更新し、**専用バックアップ `final.cite.bak.md`** に退避（`revise` の `final.bak.md`、references の `final.references.bak.md` と**名前を分ける**）。LLM 非経由・決定的。

## タスク分解

### T1. 出典ブロック生成（純関数）
- **追加**: `src/cli/inlineCite.ts`
  - `headingKey(heading: string): string`：見出しテキストから決定的にブロックキーを導出（slug もしくは hash8）。本文側の見出しからも同じ関数で再計算して突き合わせる（#1）。
  - `groupCitedSourcesByHeading(claims, sources): Map<headingText, { headingKey, sources: Source[] }>`：`present && verified` claim を **`location.heading`（テキスト）**で束ねる（anchorHash ではない。#1）。各見出しの `sourceIds`→source 解決は **`selectReferenceSources` と同じフィルタを再利用**＝**`reachable:"dead"` 除外**・primary→secondary・id 昇順・重複排除（#2。`## 参考` と同じ集合になるよう揃える）。source が0件の見出しはマップに入れない（空ブロックを作らない）。
  - `renderCiteBlock(headingKey, sources): string`：`<!-- cite:begin:<headingKey> --> … <!-- cite:end:<headingKey> -->` を返す（`> 出典: [Sxxx](url) / …`）。
  - `replaceMarkedBlocks(body, blocksByHeading): { content, applied, removedStale, skippedDuplicate }`：**多ブロック版を新規実装**（references の単一版とは別物。#4）。見出しテキスト一致で節を特定し末尾へ挿入/更新。**台帳に無い headingKey の既存 cite ブロックは除去**（stale 一掃）。**同名見出しは最初の一致のみ・2件目以降は skip（warning）**（#5）。マーカー破損（begin/end 不対応）は書き込まずエラー。
- **完了条件**: claims/sources から決定的に節別ブロック生成。見出し追加/削除/改名（headingKey 変化）で正しく追従。dead source は出さない。純関数で単体テスト可能。

### T2. CLI `article:cite`
- **追加**: `src/index.ts` に登録。`--run <id>`（必須）/ `--stdout`（書かず生成ブロック群のみ出力）/ 既定モードは profile 設定に従う。
- **挙動**: `assertRunExists` → profile の `inlineCite` が off なら**何もせず終了（理由を表示）** → claims.json/sources.json を読む（無ければ「claims-normalize を先に」エラー）→ T1 で節別 source 抽出（dead 除外済み）→ `final.md` に反映（**`final.cite.bak.md` 退避**）。対象 source が全節で0件なら書き込まず exit 1（references と同方針）。同名見出し2件目以降の skip・マーカー破損は stderr に出し、破損は中断・exit 1。
- **工数注記**: 「`replaceMarkedBlock` を一般化して流用」ではなく**多ブロックの挿入/更新/stale 除去/破損検査を新規実装**（生成側 T1・検証側 T3 とも）。単一ブロック前提の既存ロジックは extend で済まない（#4）。見積りはこの新規分を含める。
- **profile**: `--code-check` 等と同様、**run 単位で固定はしない**（決定的・再生成可能なため毎回再計算でよい）。on/off は profile 既定 ＋ CLI フラグ（`--inline-cite`/`--no-inline-cite`）で上書き可。
- **allowlist**: `templates/.claude/settings.json` に `article:cite` を追加（決定的・bak 退避ありで自動承認）。
- **完了条件**: on の profile で実行すると対象節末に出典ブロックが入り、再実行はブロックのみ更新（前後文・注記保持・stale 除去）。off の profile では no-op。

### T3. verify-artifacts の cite ブロック検査
- **変更**: [src/cli/verifyArtifacts.ts](../src/cli/verifyArtifacts.ts)
  - **多ブロック対応の cite パーサを新規追加**（既存の単一 `sources:begin/end` ロジックは流用不可。`cite:begin:<key>/cite:end:<key>` をキー対で走査。#4）。
  - cite マーカーブロック内のリンク（URL）を抽出し、**すべて `sources.json` の url 集合に含まれること**を error 検査（参考章と同じ canonicalUrl 突き合わせ）。**`reachable:"dead"` な URL がブロック内にあれば FAIL**（dead を焼かない方針を cite にも適用。#2）。
  - cite ブロックの `Sxxx` が、その**見出しの present&&verified claim の cited source 集合（dead 除外後）と一致**することを検査（手編集 drift・stale 検出）。
  - **profile は読まない方針を維持**（[verifyArtifacts.ts](../src/cli/verifyArtifacts.ts) は RunStore のみで動く。#3）。よって「cite ブロックがあれば profile に関係なく URL/ID 一致を検査する（present なら必ず validate）」とし、**「off run の取り残し warning」は profile 注入経路が要るため別フェーズ**へ送る（先の editorial-ledger ゲートと同じ制約）。
- **完了条件**: ブロック内に偽 URL/dead/不一致 ID があれば FAIL。profile に依存せず present な cite ブロックを検査。

### T4. tests ＋ ドキュメント
- **tests**:
  - `headingKey`：同一見出し→同一キー（決定的）、見出し改名→別キー。
  - `groupCitedSourcesByHeading`：**heading テキストで束ねる**・present&&verified のみ・cited source のみ・**dead source は除外（selectReferenceSources と同集合）**・primary 優先・重複排除・source 0件見出しは除外。
  - `renderCiteBlock`：行フォーマット・マーカー（headingKey 付き）・URL。
  - `replaceMarkedBlocks`：複数節への挿入／再実行で中身のみ置換／**headingKey 消滅（見出し改名/削除）で stale ブロック除去**／**同名見出しは最初の一致のみ・2件目は skip(warn)**／前後文・注記保持／破損マーカーでエラー。
  - `article:cite` CLI：profile off で no-op、claims/sources 不在でエラー、全節0件で exit 1、`--stdout` 非書き込み、書き込み時 `final.cite.bak.md` 退避。
  - `verifyArtifacts`：ブロック内偽 URL→FAIL、dead URL→FAIL、ID 不一致→FAIL、複数 cite ブロックの破損→FAIL、正常→OK（profile 非依存）。
- **docs**:
  - [docs/qiita-article-howto.md](qiita-article-howto.md) の公開前ゲート節に「（対象 profile のみ）references の**直後**に `article:cite` で本文へ見出し単位の出典を付与。**本文を変えたら references→cite を再実行**（cite は stale 化するため。#6）」。
  - [templates/.claude/agents/article-editor-in-chief.md](../templates/.claude/agents/article-editor-in-chief.md) 進行に1ステップ（profile on のときだけ）。

## スコープ外
- **文単位 inline [S001]**（特定の文へ脚注）。台帳が見出し粒度しか持たないため。実現には claim に本文オフセット/quote を持たせる別計画（factchecker 出力スキーマ拡張）が前提。
- `footnote` レンダリングモード（Qiita/Zenn 脚注記法）。初期は `marker` のみ。`inlineCite.mode` の余地だけ残す。
- 本文全リンクの strict 検査（参考リンク計画の warn 方針を踏襲）。
- LLM による出典文生成（決定的生成に固定。捏造防止）。

## 受け入れ基準
1. `article:cite --run <id>` が、対象 profile で `present && verified` claim の cited source（**dead 除外・`location.heading` で束ね**）を**見出し単位**に、本文の各節末へマーカーブロックで付与する（ブロックキー＝heading テキスト由来。`final.cite.bak.md` 退避つき・`revise`/`references` の bak を潰さない）。
2. リンク・番号は `sources.json`/`claims.json` 由来のみ（LLM に書かせない）。
3. 再実行はマーカーブロックのみ差し替え、前後文・編集注記を保持。**見出し改名/削除でキーが変わった cite ブロックは除去**（stale 一掃）。同名見出し2件目以降は skip(warn)。マーカー破損・対象0件は書き込まず exit 1。
4. profile off では no-op（off run の取り残し検出は profile 注入が要るため別フェーズ）。
5. verify-artifacts は **profile 非依存で**、cite ブロック内リンクの `sources.json` 一致（dead を含まない）と、ID 集合の台帳一致を必須（偽 URL・dead・drift を FAIL）。多ブロックの破損も FAIL。
6. `npm test` 緑・typecheck クリーン。

## 確定した論点（設計判断）
- **粒度は見出し単位**で確定。文単位 inline は台帳の制約（location は heading のみ）とコスト・脆さから不採用（別計画）。
- **既定 off・profile opt-in**で確定。一般向け読み物（hayabusa2 等）はノイズ、技術/リファレンス系で価値。
- **機械生成・機械検証・LLM 非経由**で確定（参考リンク計画と同原則。本文への手書き出典は引き続き禁止＝house style を維持しつつ、機械なら両立する）。
- **ブロック ID は見出しテキスト由来の `headingKey`**で確定（**`location.anchorHash` は claim 文ハッシュなので不可**。#1）。見出し改名でキーが変わり stale 検出が効く。グループ化キーも `location.heading`。
- **バックアップ名 `final.cite.bak.md`**で確定（revise/references と分離）。
- **生成は references の後**で確定（同じ確定済み台帳・確定本文に対して決定的に重ねる）。
- **レンダリングは初期 `marker` のみ**で確定（`footnote` は将来、profile 設定で切替）。
