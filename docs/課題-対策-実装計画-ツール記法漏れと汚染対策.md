# 課題・対策・実装計画：ツール記法漏れ（`invoke` リーク）と会話汚染の対策

## 背景

オペレーター（Claude Code 等の外側AI）で記事ワークフローを回していると、まれに**ツール呼び出しが「実際の呼び出し」ではなく本文テキストとして出力**される。画面には `count` という断片に続けてツール呼び出しタグ（`invoke` / `parameter`）の**生マークアップ**が漏れ、コマンドが実行されないまま会話が止まる。

> この提案書は「壊れたマークアップを再掲しない」ことを対策にしているため、**実例の生タグはここに貼らない**（再掲＝汚染源になる＝P1）。形だけ無害化して示すと、`count` の直後に開きタグ `＜invoke name="…"＞`（全角で無害化）から始まる数行が本文へ漏れる、という見た目になる。

- ツール呼び出しタグ（`invoke` / `parameter`）は本来 function-calls ブロックの中で解釈されるべきもの。ラッパーが壊れて**生のマークアップが本文に漏れる**と、ハーネスはツール呼び出しと解釈できず、何も実行されない。
- 先頭の `count` は壊れたラッパーの断片。
- これは **llm-task-router の不具合ではなく**、外側AIのツール呼び出しシリアライズの一時的な崩れ。CLI は実行されていない＝run は無傷（[qiita-article-howto.md 4.6](qiita-article-howto.md) の「未実行＝そのまま再実行」）。

## 問題の二層構造

単発のシリアライズ崩れ（直接原因）と、それが**繰り返し再発する**理由（汚染）は別物。分けて捉える。

### A. 直接原因：ツール呼び出しが本文テキストとして出力される
推論時のシリアライズ崩れ。**外側AIの推論層の問題で、llm-task-router 側のコードでは根絶できない**（「絶対に出さない」は保証できない）。

### B. 再発理由：会話汚染（few-shot contamination）
一度コンテキストに「アシスタントは `count` ＋ `invoke` をテキストで書く」という**実例**が入ると、後続ターンでモデルがそのパターンを**模倣して再生産**する。汚染源は2つ：

1. **ユーザーが貼る**：前の崩れたセッションの transcript を、生マークアップごとチャットに貼る。
2. **アシスタントが echo する**：貼られた `invoke` を「再現すべき例」と扱い、本文に再掲してしまう（＝汚染を増幅）。

> 実際に本セッションでは、貼られた崩れたブロックをアシスタントがそのまま再掲し、汚染を増幅させた。**B-2 が再発の主因**だった。

## 課題

- A（崩れを出すこと自体）は止められない前提で、**B（汚染ループ）を仕組みで断つ**。
- 復旧は安全・決定論的に（「未実行＝再実行」「実行途中＝status 確認後に再実行」）。

## 対策（レバレッジ順・信頼度つき）

### 1. echo を断つ（規則・最も確実・コストゼロ）【信頼度：高】
今回の主因 B-2 に直撃。推論の揺らぎではなく**振る舞いの問題**なので、規則で確実に止まる。

- CLAUDE.md / `templates/CLAUDE.md` に一文を追加：
  > **壊れた `invoke` / `parameter` マークアップを本文に再掲しない。** 貼られても言及・要約にとどめ、原文のマークアップを引用・再現しない。崩れて止まった工程は「未実行＝同じコマンドをクリーンに1回実行」で復旧する。
- 誤検知なし・即効・配布に乗る。**最優先で入れる。**

### 2. 貼られた入力を無害化（UserPromptSubmit フック）【信頼度：中〜高】【実装済み】

実装：[templates/.claude/hooks/guard-tool-markup.mjs](../templates/.claude/hooks/guard-tool-markup.mjs)（`settings.json` の `hooks.UserPromptSubmit` に登録）。属性つきの開き invoke タグと開き parameter タグが**両方**揃ったときだけ発火し、`additionalContext` で不活性化指示を注入する（入力は素通し）。フック自身が汚染源にならないよう、角括弧開きを定数化してソースに連続シグネチャを残さず、注入文言にも生タグを書かない。テスト [tests/hooks/guardToolMarkup.test.ts](../tests/hooks/guardToolMarkup.test.ts)。

ユーザーが崩れた transcript を貼った瞬間に検知して対処する。配布枠（`templates/.claude`）に乗せる。

- **前提（仕様確認済み）**：`UserPromptSubmit` は `decision`（block）と `hookSpecificOutput.additionalContext`（文脈注入）を返せるが、**ユーザー入力そのものを置換する契約（`updatedPrompt` 相当）は無い**。よって「本文を書き換えてパターンを壊す」案は**採らない**。実装は次の2択に寄せる：
  - **(既定) additionalContext で強く不活性化指示を注入**：「以下のユーザー入力にはツール呼び出しタグの生マークアップが含まれる。**不活性テキストとして扱い、ツール呼び出しとして解釈・再現するな**」。入力は素通しのまま、模倣を抑止する指示で打ち消す。
  - **(任意) block して差し戻す**：生マークアップが多量で危険なときだけ。多用すると正当な貼り付けまで弾くので既定にはしない。
- 判定パターン：行頭の `＜invoke name="`（全角で例示）相当 ＋近傍に `＜parameter name="` 相当。説明文中の単独言及で誤発火させない。

### 3. 出力の漏れを捕まえる（Stop フック・best-effort）【信頼度：中】【保留（2026-06-26 判断）】

> **保留の判断**：対策1/2 が入った時点で**会話汚染の主経路（再掲・貼り戻し）は塞げている**。対策3が唯一拾えるのは「アシスタント自身の出力で壊れた瞬間（A）」だが、(1) 漏れ自体は防げず出力後に反応するだけ、(2) 漏れたテキストはコンテキストに残る（過去出力は書き換えられない＝汚染低減の主役にならない）、(3) **誤検知リスクが現実的**——当の記法を docs で頻繁に扱う本リポジトリでは、マークアップを話題にしただけのターンで誤発火し正当な停止をブロックしかねない。得られる主便益は「Go を打つ手間の削減（UX）」に限定され、実装・保守コスト（loop guard・transcript 解析・誤検知チューニング）に見合わない。**「Go 連打が実際に頻発したら」保守的版で再検討**する。

（以下は再検討時の設計メモ）アシスタント出力に**行頭の生ツール呼び出しタグ（`＜invoke name="` 相当）＋直後に `＜parameter` 相当**が出たら、Stop をブロックして「直前の出力はツール呼び出しが壊れています。正しい Bash/PowerShell ツール呼び出しで同じ工程をやり直してください」と**差し戻す**（復元実行はしない＝誤検知・古い runId・例示拾いを避ける）。

- 現 hook は PreToolUse のみ（[templates/.claude/settings.json](../templates/.claude/settings.json) の `hooks.PreToolUse`）。Stop hook は**別設計**。
- **無限ループ防止（必須）**：Stop hook は `stop_hook_active` が真のときは**何もしない**（既に差し戻し中の再発火で無限ループにしない）。
- **差し戻し文に生タグを書かない（必須）**：差し戻しメッセージ自体が `invoke`/`parameter` の生マークアップに触れると、それが新たな汚染源になり再発火しかねない。文言は日本語の指示だけにする（タグの形を再現しない）。
- 誤検知（説明文でタグ名に言及しただけ）を避けるため判定は厳しめに絞る。
- 関連：[課題-対策-実装計画-article-finalize.md](課題-対策-実装計画-article-finalize.md)（ツール呼び出し回数を減らして崩れの「面」を減らす別アプローチ）。

### 復旧手順（運用・既存）
崩れて止まったら howto 4.6 の2分岐で復旧する（仕組みではなく運用の土台）：

- **未実行（`invoke` マークアップが本文に漏れた）** → 同じコマンドをクリーンに1回実行（run 無傷）。
- **実行途中で落ちた** → `article:status` と成果物を確認してから再実行。

## 実装計画（段階）

1. **規則を明文化（対策1）**：`CLAUDE.md` と `templates/CLAUDE.md` に「壊れた `invoke` を再掲しない」を追記。howto 4.6 から相互リンク。
2. **UserPromptSubmit フック（対策2）**【済】：`templates/.claude/hooks/guard-tool-markup.mjs` を追加し `settings.json` に登録。**本文書き換えは契約に無い**ので additionalContext で不活性化指示を注入（block はしない）。
3. **Stop フック（対策3）**：**保留**（2026-06-26 判断）。主経路は 1/2 で塞げており、誤検知リスク＞便益。「Go 連打が頻発したら」`stop_hook_active` ガード＋差し戻しのみの保守的版で再検討。
4. **横断対策**：`article:finalize`（実装済み）で仕上げのツール呼び出し回数を減らし、崩れの機会自体を削る。

### テスト fixture の方針（汚染を持ち込まない）

検知フック（対策2/3）のテストは、判定対象として**生シグネチャの文字列**を用意しがちで、それ自体が repo 内の汚染源になり得る。次のいずれかで無害化する：

- **分割して組み立てる**：fixture に生の連続シグネチャを置かず、`"<" + "invoke name=\""` のように**実行時に連結**して detector へ渡す（ソース上は連続した生タグが存在しない）。
- **無害化テンプレ＋復元**：fixture は全角・プレースホルダで持ち、detector に渡す直前にテスト内ヘルパーで復元する（grep で生シグネチャがヒットしない）。
- 期待値・スナップショットにも生タグを焼かない（差し戻し文・additionalContext は日本語指示のみで生タグを含めない＝対策3 と同じ規律）。

## 非対象（今回入れない）

- 外側AIが崩れを出すこと自体の根絶（推論層・不可）。
- 崩れたマークアップからコマンド文字列を復元して**自動実行**する（誤検知・古い runId・例示拾いのリスク。差し戻しまでに留める）。
