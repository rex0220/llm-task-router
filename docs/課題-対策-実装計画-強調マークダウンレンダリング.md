# 検討メモ：強調 `**…**` がレンダリングされない問題（CJK フランキング）

> ステータス: **しくみの検討のみ**（実装・PR は未着手）。本書は原因・対策案・効果・リスクの整理に閉じる。

課題: 生成記事の本文で `**…**`（strong 強調）の `**` が**そのまま文字として表示**され、太字にならないケースがある。Qiita（markdown-it / CommonMark 準拠）で再現。実データ: `export/hayabusa2.md` で5箇所。これは LLM の記法ミスではなく、**CommonMark の強調デリミタ判定（フランキング規則）が CJK 約物と相性が悪い**ことに起因する構造的な問題。

---

## 原因

CommonMark では `**` が強調を**閉じる**には、その `**` が **right-flanking デリミタ列**である必要がある。条件は：

- 直前が空白でない、かつ
- **(a)** 直前が約物でない、**または** **(b)** 直前が約物で、かつ直後が空白か約物

`」` `）` `”` 等は Unicode 句読点カテゴリ（約物）。これらが**閉じ `**` の直前**にあり、**直後が普通の日本語文字**（空白でも約物でもない）だと、(a)(b) いずれも不成立 → right-flanking にならず、`**` が閉じデリミタとして無効化され、強調が成立しない。

**開き側も同じ理屈で壊れる。** `**` が強調を**開く**には left-flanking デリミタ列である必要があり、条件は対称（直後が空白でない、かつ「直後が約物でない」または「直後が約物で直前が空白か約物」）。たとえば `これは**「太陽系」**の` では、開き `**` の直前が通常文字 `は`・直後が約物 `「` のため left-flanking にならず、**開けない**ことがある。今回の実データ5箇所は「閉じ側」だが、原因・検出・規約はいずれも**「開けない `**`」と「閉じられない `**`」の両方**を対象に揃える必要がある。

英語では単語間に空白があり境界を空白が担うが、日本語は単語間に空白がなく、約物が境界に立つため発火しやすい。**日本語 × 約物に固有の構造問題**であり、プロンプトでいくら注意しても確率的に再発する。

### 実データ（`export/hayabusa2.md`）

| 行 | 記述（抜粋） | 閉じ `**` 直前 | 直後 | 結果 |
|---|---|---|---|---|
| 34 | `**「太陽系の化石」**の` | `」` | `の` | 太字にならない |
| 83 | `…“次の世代の再挑戦”**でした` | `”` | `で` | 同上 |
| 344 | `**SCI（…衝突装置）**を用いて` | `）` | `を` | 同上 |
| 436 | `**約5.4g（…ごく少量）**と公表` | `）` | `と` | 同上 |
| 437 | `**100mg（0.1g）**を大きく上回る` | `）` | `を` | 同上 |

いずれも「閉じ `**` の直前が約物・直後が日本語文字」という同一パターン。一方 `**初代はやぶさ**を`（直前が文字 `さ`）は (a) を満たし正常に閉じる。

### 比較: 正常にレンダリングされる例（同記事内）

```markdown
…まず外せないのが**初代はやぶさ**です。      閉じ ** の直前が「さ」(文字) → 正常に太字
- **成功**：遠い小惑星まで行き…              直後が「：」だが開き側・閉じ直前が文字 → 正常
**2018年6月**、はやぶさ2はついに…           閉じ直前が「月」(文字) → 正常
```

この実データ（いずれも閉じ側）では差は一点のみ: **閉じ `**` の直前が「文字」か「約物」か**。約物（`」` `）` `”`）だと崩れ、文字なら成立する。なお開き側でも同型の崩れが起きる（開き `**` の直後が約物のケース。原因節を参照）。

## 根本のルール（記述規約）

**`**` の内端（開き直後・閉じ直前）に約物を置かない。括弧・引用符・読点は `**` の外に出す。**

```
× 、**「太陽系の化石」**の      内端が 「 」（約物）
○ 、「**太陽系の化石**」の      内端が 太 石（文字）

× **約5.4g（…少量）**と         内端が 約 ）（約物）
○ **約5.4g**（…少量）と         内端が 約 g（文字）
```

これを守れば、CommonMark 系レンダラでは周囲の文字に関わらず安定して成立しやすい。要約: **「括弧・引用符・読点は太字の外、太字の中は文字で始まり文字で終わる」**。

---

## 対策案

予防（確率的）と保証（決定的）の2層で考える。本リポジトリの既存思想（プロンプト指示＋ `verify-artifacts` の決定的ゲート）に沿う。

### 第1層: 生成時ルール（予防・確率的）

`write-article` / `article:create` ・ `refine` のプロンプトに上記の記述規約を明記する。発生頻度は下げられるが LLM 由来のため**取りこぼしは残る**。単独では「仕組み上問題ない」を満たさない。

### 第2層: 機械ゲート（保証・決定的）← 本命

`final.md` を走査し、**開けない／閉じられない `**` を検出**したら公開前ゲートで報告する。差し込み先は既存の決定的ゲート [src/cli/verifyArtifacts.ts](../src/cli/verifyArtifacts.ts)。判定エンジンは2案：

| 案 | 方式 | 長所 | 短所 |
|---|---|---|---|
| **A** | 無依存の文字列 lint。各 `**` の前後文字を「空白/約物/その他」に分類し、CommonMark フランキング規則（開きは left-flanking・閉じは right-flanking）で開閉可否を判定。**開けない `**`／閉じられない `**`** の両方を行番号付きで報告。約物判定に Unicode 句読点カテゴリ（`」` `）` `”` 等）を使う | 既存 repo 思想（無依存・正規表現処理: `stripWrappingCodeFence` / `detectWrapText` と同様）に合致。軽量・高速 | フランキング規則の該当部分を自前実装する（仕様追従の保守コスト） |
| **B** | `markdown-it`（Qiita に近い CommonMark パーサ）を導入し、レンダ後のテキストノードに `**` が残れば壊れと判定 | パーサがコード領域を区別するので誤検知が少なく実装が薄い | 新規依存を追加（現状マークダウンパーサ依存なし）。**Qiita 完全一致ではない**（Qiita 側の markdown-it バージョン・プラグイン差分までは固定できない）。正当なエスケープ `\*\*` をレンダ後テキストの `**` として誤検知し得る |

> 注意（重要）: **`verify-artifacts` を通しても、それだけでは export はブロックされない。** 現状 `article:export`（[src/index.ts:332](../src/index.ts)）は `exportFinalArticle` を呼んで書き出すだけで、`verifyArtifacts`（[src/index.ts:403](../src/index.ts)）は別コマンド。公開ブロックは**手順上の運用**（編集長が export 前に verify を回す）であって、コードによる強制ではない。lint を verify-artifacts に足しても、それは「公開前ゲートで検出できる」までで、「export 不可を保証」にはならない。保証まで行くなら次の2点が追加で要る（→「保証に必要な追加設計」節）:
> 1. **export 時に verify を再実行**するか、**verify 済み `final.md` の hash を記録し export 時に一致確認**する（古い OK のまま export を防ぐ）。
> 2. lint 結果を `error`（FAIL）に格上げする。

### 第3層: 自動修正（任意・opt-in）

「約物を `**` の外へ動かす」変換は**強調語の文字列（地の文）を変えない**ため `--fix` 相当で機械適用が原理的に可能。ただし**どこからどこまでを太字にするかという強調範囲の意図は変わる**（例: `**「太陽系の化石」**` → `「**太陽系の化石**」` でかぎ括弧が太字から外れる）。地の文は不変でも「筆者がどこを強調したかったか」を機械が勝手に再解釈し得るため既定オフ・確認付きが前提。本検討では**範囲外（後追い）**とする。なお本文修正は CLAUDE.md の原則上 `final.md` 直接編集ではなく `article:revise` 経由で戻すのが正路。

## 効果

- **対象パターン（`**` の約物内端で開閉できないケース）は、仕様どおり実装できれば公開前ゲートで決定的に検出できる**。これが「仕組み上問題ない」の核。ただし検出力は **A 案の実装精度（CommonMark フランキング・Unicode 句読点分類・エスケープ `\*`・コード領域除外）に依存**する＝「無条件に必ず」ではなく「**仕様を正しく実装した範囲で決定的**」。さらに**全 Markdown 記法の崩れを保証するわけではない**（`*` イタリック・`__`・`***`・入れ子は別問題として残る。下記リスク参照）。「取りこぼしゼロ」ではなく「**対象パターンを仕様どおり決定的に検出**」が正確な表現。
- 「export 不可を保証」まで行くには、ゲートを `error` に格上げし、かつ export 側で verify 再実行／hash 鮮度確認を足す必要がある（第2層の注意・後述の追加設計）。lint 単体では「検出」までで「公開ブロック」は別途。
- 検出は行番号付きなので、編集長が `article:revise` で直す箇所が一意に分かる。
- 第1層併用で revise の往復回数自体も減る（発生を未然に抑制）。
- 既存の `verify-artifacts` に同居するため、新しい運用ステップを増やさない（編集長が公開前に回す既存ゲートの一部に乗る。export からの自動実行は現状なし）。

## リスク・留意点

- **誤検知（false positive）**: 本文中の正当なリテラル `*`／エスケープ `\*\*`（数式・コード断片・脚注記号など）を強調崩れと誤判定する恐れ。コードフェンス／インラインコード内は除外し、エスケープ `\*` も非デリミタとして扱う設計が必須。A 案はここを自前で正しく扱う必要がある。B 案はパーサがコード領域を区別するので比較的安全だが、レンダ後テキストに `**` が残る方式だと**正当なエスケープ `\*\*`（`**` 表示が目的）まで検知し得る**。
- **取りこぼし（false negative）**: A 案はフランキング規則の実装精度に依存。検出は**開き側（left-flanking 不成立）と閉じ側（right-flanking 不成立）の両方**を見る必要がある（閉じ側だけだと `これは**「太陽系」**の` の開き側崩れを逃す）。`__`（アンダースコア強調）・`*`（イタリック）・入れ子・3連 `***` 等まで網羅するとロジックが増える。最初は `**` の約物内端ケースに限定し、対象を明示するのが安全。
- **依存追加（B 案）**: 現状マークダウンパーサ依存ゼロの方針を崩す。サプライチェーン/メンテ面のトレードオフ。
- **強制レベルの判断**: `error`（公開ブロック）にするか `warning`（検知のみ、`detectWrapText` と同様）にするか。ブロックにすると保証は強いが、稀な正当リテラル `**` で詰まる可能性 → エスケープ（`\*\*`）やコード化の回避手段を案内できるようにする。
- **既存記事への遡及**: ゲートを `error` で入れると、過去 run の再 export 時に既存の崩れが顕在化して FAIL し得る。導入時は warning から始める／対象を新規 run に限る等の移行設計が要る。

## 保証に必要な追加設計（lint だけでは閉じない穴）

第2層の lint を入れても、それだけでは「壊れたまま公開」を**コードで防げない**。理由は2つ:

1. **export が verify を呼ばない**: `article:export`（[src/index.ts:332](../src/index.ts)）は書き出すだけ。lint を verify-artifacts に足しても、編集長が verify を回さずに export すれば素通りする。
2. **verify 結果が `final.md` の版に紐づかない**: 仮に verify を通しても、その後 `article:revise` や `article:references` で `final.md` が変わると、**古い OK のまま export** できてしまう（verify と export の間に本文が変わる窓）。

→ 「export 不可を保証」するには、次のいずれかが必要:

- **(i) export 時に再検証**: `article:export` の中で検証を実行し、FAIL なら書き出さない（`--force` 等の明示オーバーライドは別途）。最も確実。
- **(ii) hash 鮮度確認**: verify 実行時に検証メタを記録し、export 時に現在の `final.md` hash と一致するときだけ許可。一致しなければ「verify を回し直せ」と促す。

#### 論点 A: (i) で「何を」再実行するか — full `verifyArtifacts` か 強調 lint 単体か

これが**この設計の最大の論点**。export 強制に full `verifyArtifacts` を使うと blast radius が大きい: `verifyArtifacts` は強調 lint だけでなく claims / sources / editorial-ledger / publication-check まで全部見るため、「Markdown レンダリング不備を防ぐ」目的を超えて、無関係なゲート未了でも export が止まる。

- **full `verifyArtifacts` を呼ぶ**: 公開前ゲート全体を export 直前に強制でき一貫性は高いが、強調以外の理由での停止が増える（編集長の手順ゲートと二重化・重い）。
- **強調 lint 単体を呼ぶ**: 目的（レンダリング不備）に対して最小。blast radius が小さく `--force` 設計も単純。ただし「export 直前ゲート」が強調だけ特別扱いになる非対称が生じる。

→ **未決事項に格上げ**（下記 項5）。本検討の推奨は「目的に最小な**強調 lint 単体**を export 直前に置く。full verify は従来どおり編集長の手順ゲートに残す」。

#### 論点 B: (ii) で「何を」記録するか

hash だけだと、**どの厳しさ・どのルールで通した OK なのか**が後から分からない（`warning` 時代に通したものか、`error` 化後か、どの lint ルールセットか）。構造化して残す:

```ts
// runs/<id>/markdown-lint-stamp.json（例）
{
  finalHash: string;     // 検証時の final.md の hash（hash.ts 流用）
  ruleVersion: string;   // lint ルールセットの版（対象パターン定義の変更を追える）
  severityMode: "warning" | "error";  // どの強制レベルで通したか
  result: "pass" | "fail";
  verifiedAt: string;    // 検証時刻（YYYY-MM-DD など既存粒度に合わせる）
}
```

export 時は `finalHash` 一致に加え、**`severityMode==="error"` かつ `result==="pass"`** を要求する（warning 時代の pass を保証扱いにしない）。`ruleVersion` 不一致なら「新ルールで再検証せよ」と促せる。

**(i) と (ii) は排他ではなく併用可能。** (i) は「export 時に検証を回す（古い OK を防ぐ実行ゲート）」、(ii) は「いつ・どの厳しさ・どのルールで通したかを残す監査スタンプ」で役割が違う。(i) は実装が単純で確実だが export に検証の重さが乗る。(ii) は責務分離を保つが「verify→export の間に確認」という新しい不変条件を1つ増やす。**本検討の推奨は併用**: export 直前に **(i) 強調 lint 単体を再実行**しつつ、**(ii) 構造化スタンプを監査用に残す**（lint 単体運用でも `severityMode`/`ruleVersion` の記録は格上げ判断・再検証要否に効く）。

## 実装計画（A 案・段階導入・着手時の叩き台）

1. **Markdown 走査ユーティリティを追加**（[src/utils/text.ts](../src/utils/text.ts) に同居 or 新規）。コードフェンス（``` ```）・インラインコード（`` ` ``）・エスケープ `\*` を除外して `**` デリミタ列を取り出す。
2. **`**` の約物内端ケースに限定**して開閉判定。**開けない `**`（left-flanking 不成立）と閉じられない `**`（right-flanking 不成立）の両方**を行番号付きで返す。約物は Unicode 句読点カテゴリで判定。`*`・`__`・`***`・入れ子は対象外（明示）。
3. **`verifyArtifacts` に `warning` として追加**から開始（`detectWrapText` と同じ非ブロック扱い）。`error` 格上げの根拠は5例では不足。**格上げ条件は false positive と false negative の両面**で揃える:
   - **false positive 0（誤検知なし）**: `export/`・`runs/*/final.md`・`docs/` の一括走査、コードを多く含む記事、数式風記述（`a*b`・`*ptr` 等）、エスケープ `\*\*` を含むサンプル、Qiita 想定記事数本で、正当な記述を1件も誤検知しない。
   - **golden test 通過（検出漏れなし）**: 今回の5例（閉じ側）と開き側失敗例（`これは**「太陽系」**の`）を**必ず検出**し、既知の正常例（`**初代はやぶさ**` 等）を**誤検出しない**ことを固定テストで保証。
   両方を満たしてから `error` へ格上げ。
4. **保証まで行くなら**「保証に必要な追加設計」の (i) export 時再検証（論点 A: 強調 lint 単体 / full verify）か (ii) hash 鮮度確認（論点 B: 構造化スタンプ）を追加し、ゲートを `error` 化。
