# 実装計画：進捗ログ基盤（優先度1）

対象: [docs/課題-対策案.md](課題-対策案.md) の優先度1 — **進捗の正本 `progress.events.jsonl` → `progress.json` → `progress.md` ＋ `article:status`**（課題1・4・7 をまとめて回収）。

この文書は実装計画のみ（コードは含まない）。LGTM 済みの対策案を、着手できるタスクへ分解する。

## 設計の確定事項（実装中に揺らさない）

- **正本は append-only の `runs/<runId>/progress.events.jsonl`**（唯一の source of truth）。1イベント=1行 JSON。再実行・`resume`・途中失敗でも追記だけなので壊れにくい。
- **`progress.json` は events からの機械可読スナップショット（派生）**、**`progress.md` は表示用スナップショット（派生）**。どちらも正本ではなく、events から再生成する。
- **`meta.json.steps` は既存互換/補助**（現行コードとの後方互換のために残す。進捗の判断には使わない）。
- **書き込み経路は1本に集約**。CLI もサブエージェントも `progress.events.jsonl` への追記（`progress:event` ヘルパ）経由でのみ記録。**サブエージェントに `progress.json` を直接書かせない**。
- **コスト**: `progress.json` のスナップショットに `steps[].costUsd` ＋ `totalCostUsd` を集約値として持つ（元データは events の costUsd）。`cost.json` は作るなら派生扱い。
- **再生成タイミング**: events が `progress.json` より新しければ「読む直前に再生成」。各イベント後の即時再生成は任意の最適化。
- **`error` イベントの記録方針（誰がいつ打つか）**: 各 CLI action の `try/catch` で、失敗時に対象 step の `status:"error"`（`note` に短い理由）を打ってから例外を再送出する。サブエージェントは異常終了時に `progress:event --status error` を打つ運用。**集約では `error` は `done` を上書きしない**（リトライ成功＝後続の `done` を優先。T1 の状態優先度に従う）。記録のテストは done/skip/error の3系統を最小ケースで持つ。

## タスク分解

### T1. progress の型・集約ロジック追加
- **追加**: `src/progress/types.ts`（`ProgressEvent` / `ProgressStep` / `ProgressSnapshot`）。
  - `ProgressEvent`: `{ at, runId, step, status: "start"|"done"|"skip"|"error", task?, provider?, model?, elapsedMs?, costUsd?, output?, note? }`
  - `ProgressStep`（集約後）: `{ step, status, startedAt, finishedAt, elapsedMs, costUsd, output }`
  - `ProgressSnapshot`: `{ runId, steps: ProgressStep[], totalCostUsd, currentIndex, total, updatedAt }`
- **追加**: `src/progress/stepOrder.ts` — **canonical な工程順**（`currentIndex`/`total` を安定させるため）。
  - イベントが来た順に並べると「7工程中何番目」が不安定になる。profile/workflow ごとの**標準工程定義**（例: 企画 → create/refine → final-review → factcheck＋build-verify → editorial → claims-normalize → publication-check/verify-artifacts）を持ち、`total` と各 step の index はここから決める。
  - canonical に無い step（アドホックなサブエージェント記録等）は末尾に「追加工程」として並べ、`total` には数えるが順序は登場順。
- **追加**: `src/progress/aggregate.ts` — events 列 → `ProgressSnapshot` への純関数 `aggregate(events, stepOrder): ProgressSnapshot`。
  - 同一 step は後勝ち upsert（start→done で1ステップに畳む）。**ただし `skip` は完了済み（`done`）を劣化させない**。集約ルール:
    - `done` は最終状態（以後の `skip` で上書きしない。resume 時に完了済み工程へ来る `step:skip` で `done`→`skip` に退行させない）。
    - `error` は `done` を上書きしない限定。`error` 後に同 step の `done` が来たら `done`（リトライ成功）。
    - 状態優先度: `done` > `error` > `skip` > `start`。同 step は最高優先度・同優先度後勝ちで状態確定（`startedAt` は最初の start、`finishedAt`/`note`/`output`/`provider`/`model` は最終状態を確定させた代表イベント＝最後に状態を更新したイベントを採用。skip/error の note が done 行に混ざらないようにするため）。
    - **`elapsedMs`/`costUsd` は代表イベントからではなく全イベントの積算**（同一 step に複数回 done が来ても後勝ちで上書きしない）。`create`→`resume`→`review` のように同じ canonical step を複数 action が跨ぐケースで、後発の安価な action が先発のコストを消さないようにするため。
  - `currentIndex` は「未完（done/skip でない）の最小 canonical index」、なければ最終工程。
  - `totalCostUsd` は costUsd が定義された step のみ合算（不明は除外。既存 `utils/cost.ts`・refine summary と同方針）。
- **完了条件**: 純関数として単体テスト可能（I/O なし）。stepOrder を渡せば `currentIndex`/`total` が安定。

### T2. 記録ヘルパ（書き込み経路の一本化）
- **追加**: `src/progress/RunProgress.ts`
  - `appendEvent(runId, event)`: `runs/<runId>/progress.events.jsonl` へ1行追記（`mkdir -p` 込み。`RunLogger` と同じ append パターン）。
  - `regenerate(runId)`: events を読み → `aggregate` → `progress.json` と `progress.md` を書き出す。
  - `readSnapshot(runId)`: events が json より新しければ `regenerate` してから `progress.json` を返す（「読む直前に再生成」）。
- **追加**: `src/progress/renderMarkdown.ts` — `ProgressSnapshot` → 表 Markdown（列: `# / 工程 / 状態 / 開始 / 終了 / 所要 / 概算$ / 根拠`）。課題.md の「7工程中6番目」表に対応。
- **追加（必須）**: `article:progress:event` サブコマンド（CLI/サブエージェント共通の入口）。**サブエージェント記録をスコープに含める以上、これは必須成果物**（直接 JSON を触らせず、書き込みを1経路に集約するため）。`--run <id> --step <name> --status <start|done|skip|error> [--note ...]` 等を受ける。
- **完了条件**: 追記は並行でも行が壊れない（append-only）。regenerate は冪等。`progress:event` から記録 → `readSnapshot` で反映を確認できる。

### T3. 既存イベント（WorkflowEvent と RefineEvent の両方）を progress に接続
- **対象を書き分ける**:
  - `article:create` / `resume` / `review` / `evaluate` / `revise` は `WorkflowEvent`（`createProgressReporter`、[src/index.ts](../src/index.ts)）。
  - `article:refine` は **`RefineEvent`**（`createRefineReporter`、[src/index.ts](../src/index.ts) 付近）。`WorkflowEvent` ではない。両 reporter に接続が要る。
- **⚠ 同期 reporter のままだと fire-and-forget でイベントが落ちる**。現在の `WorkflowReporter` / `RefineReporter` は同期関数で、`appendEvent`（async）を投げっぱなしにするとプロセス終了時に追記が flush されない恐れ。
  - **採用した方式（action 側で集約 → flush）**: reporter は内部の `step:*` イベントを集計するだけ（同期、`totals()` で costUsd/elapsedMs の合計を返す）。**`progress.events.jsonl` への記録は内部イベント単位ではなく、CLI action 単位**で行う：共通ヘルパ `runWithProgress` が action 開始時に `start` を1件、action 終了時に `done`（`totals()` の集約値＋出力パス）または `error`（`shortMessage(error)`）を1件、確実に `await` して書く。途中失敗時も `error` 記録後に例外を再送出。
  - refine の round 単位の内訳（round:start/evaluation/revision 等）は `progress.events.jsonl` には記録しない。round 内訳は既存の `meta.json` の `refine.rounds` / stderr 表示で確認する（progress はあくまで「7工程の現在地」を見るためのもので、round 粒度までは持たない）。
- **同じ canonical step に複数回 done が来る場合の集計**: `create` → `resume` → `review` のように同じ `step` に対して action 単位の `done` が複数回記録され得る。状態（status）は最高優先度・同優先度後勝ちだが、**costUsd/elapsedMs は後勝ちで上書きせず全イベント分を積算**する（[T1](#t1-progress-の型集約ロジック追加) のルールに合わせる）。これをしないと、costのかからない `resume`（やり直し不要で即 done）が `create` 本体のコストを消してしまう。
- **注意**: stdout（runId/最終パス）は変えない。記録は副作用として足すだけ。
- **完了条件**: create→refine→evaluate→revise を通すと `progress.events.jsonl` に対応行が**確実に**残る（プロセス終了で欠落しない）。

### T4. CLI 終了イベントの記録（7工程を覆う）
- **対象（canonical 工程に対応する CLI に限定）**: `claims-normalize` / `verify-artifacts` / `review-editorial`(editorial) / `export`。`WorkflowEvent` を持たないが7工程本体に入るもの。
- **対象外（意図的に記録しない）**: `import` / `update-diff` / `claims-recheck` は更新リライト系・補助工程で canonical 工程に入らないため記録しない（記録すると末尾に「追加工程」として並びノイズになる）。必要になったら canonical 定義に追加してから対応する。
- **変更**: 各コマンドの正常終了/スキップ時に `appendEvent`（成功 status、出力パス、blocking 件数などを `note` に）。
- **対象外（サブエージェント運用）**: factcheck / build-verify / publication-check は CLI を持たないため、**`article:progress:event` 経由で編集長/サブエージェントが記録**する運用ルール。`.claude/` の手順とサブエージェント定義に「終了時に progress:event を打つ」を追記（T6 と連動）。
- **完了条件**: 7工程すべてが `progress.json` に現れうる（CLI 由来＋イベント記録ルール）。

### T5. `article:status` 追加
- **追加**: `src/cli/status.ts` ＋ `src/index.ts` にサブコマンド登録。
- **挙動**: `article:status --run <id>` で `readSnapshot`（読む直前に再生成）→ 現在地表＋ `total` 進捗＋ `totalCostUsd` を表示。
  - 既定は人間向け表（`progress.md` 相当を stdout）。`--json` で `ProgressSnapshot` を出す（スクリプト用）。
- **完了条件**: 課題.md の「現在地：7工程中6番目」を1コマンドで再現できる。

### T6. tests ＋ 配布物の更新
- **tests（`vitest`）**:
  - `aggregate`: start/done 畳み込み、再実行で重複しない（後勝ち）、totalCost は不明除外。
    - **状態優先度の退行防止**: resume の `done`→`skip` で `done` が劣化しない。`error`→`done`（リトライ成功）で `done` になる。状態優先度 `done > error > skip > start` の確定。
    - **canonical 工程順**: stepOrder を渡すと `currentIndex`/`total` が登場順に依存せず安定。canonical 外 step は末尾に並ぶ。
  - `RunProgress`: 追記→regenerate の冪等、events が新しいとき readSnapshot が再生成。
  - `renderMarkdown`: 表の列・n/a 表示。
  - `progress:event`: 記録 → readSnapshot で反映（done/skip/error の3系統）。
  - `status`: `--json` 出力スナップショット。
- **配布/ドキュメント**:
  - `.claude/settings.json`（template）: `article:status` / `article:progress:event` を allowlist に追加（記録系は承認不要に。export/build-verify はプロンプト維持）。
  - サブエージェント定義（factchecker / build-verifier / editor-in-chief）に「工程終了時に `progress:event` を打つ」を追記。
  - 手順書 [docs/qiita-article-howto.md](qiita-article-howto.md) に `article:status` と progress の章を追加。
- **完了条件**: `npm test` 緑。`init` 後のフォルダで `article:status` がプロンプトなしで動く。

## 依存関係と着手順

```
T1 (型・集約) ─┬─ T2 (記録ヘルパ) ─┬─ T3 (既存イベント接続)
              │                   ├─ T4 (CLI終了イベント)
              │                   └─ T5 (article:status)
              └──────────────────── T6 (tests・配布)  ← 各タスクと並行で足す
```

- **T1 → T2 が土台**。ここが固まれば T3/T4/T5 は並行可能。
- 最小で価値が出る縦切り: **T1 → T2 → T3 → T5**（create〜revise の進捗が `article:status` で見える）。T4/T6 はその後で7工程フル対応に広げる。

## 受け入れ基準（優先度1 全体）

1. create→refine→evaluate→revise を通すと `runs/<id>/progress.events.jsonl` と再生成された `progress.json`/`progress.md` が残る。
2. `claims-normalize` / `verify-artifacts` 実行で対応イベントが追記される。
3. `article:status --run <id>` が現在地表・進捗・概算合計を表示し、`--json` でも出せる。
4. 再実行・`resume` で行が重複しない（後勝ち集約）。
5. サブエージェントは `progress:event` 経由でのみ記録（直接 JSON 書き込みなし）。
6. `npm test` 緑、`article:status`/`progress:event` がプロンプトなしで動く。
