# 振り返りと実装計画：公開物への死リンク混入の再発防止

課題: **export 済み・全工程 GO の公開相当物（jomon シリーズ全6回）に、参考章の死リンクが計7件（4記事）混入**していた。ユーザーからの1件の指摘（`http://www.maibun.or.jp/`）を起点に発覚し、横断点検で残り6件も判明した。本ドキュメントは、何が起き・どう復旧し・なぜ通り抜けたかを記録し、既存の到達性メタ（[課題-対策-実装計画-source到達性メタ.md](課題-対策-実装計画-source到達性メタ.md)）と `article:sources-check`（[課題-対策-実装計画-sources-check.md](課題-対策-実装計画-sources-check.md)）を土台に、再発防止のしくみ改善案をまとめる。

対象データ: `llm-router-e2e/runs/2026-06-24-jomon-*`（生成ツール 0.2.53）。

## 2. 今回の対策（復旧手順）

「LLM に URL を書かせない＝偽 URL 防止」の原則を保ったまま復旧した。

1. **横断点検**: `article:sources-check --run <id> --only-cited` を全6 run に実行 → dead/unknown を抽出。
2. **実体確認**: 各 URL を `curl -L`（ブラウザ UA）＋ `nslookup` で直接確認し、機械判定の取りこぼし（maibun=NXDOMAIN なのに unknown 等）と誤検出候補（timeout/接続拒否）を切り分け。
3. **差し替え（生存代替なし3件）**: `article-factchecker` に同一事実を裏取りする **到達確認済み（WebFetch 200）の代替 source** を探させ、`sources.raw.json` を同一 key のまま url/title/summary/sourceType 更新（`reachable:"ok"`/`checkedAt`）。
   - intro 弥生開始 → 歴博「IntCal20較正曲線…」
   - crafts アスファルト → 世界遺産「北海道・北東北の縄文遺跡群」公式
   - crafts 抜歯 → 藤井寺市教育委員会 文化財保護課
4. **除去（生存代替あり4件）**: `sources.raw.json` の当該 source を `reachable:"dead"` にし、`claims.raw.json` の該当 claim の `sourceRefs` から死 key を除去（各 claim に生存出典が1件以上残ることを確認）。
5. **再生成**: `article:claims-normalize` → `article:references`（参考章を機械再生成）→ `article:verify-artifacts`（**4記事すべて OK**）→ `article:export --force`。
6. **検証**: 死リンク7件が全 export から消滅・差し替え先は全件 200・全6記事 OK。

> 副次的に判明: 再 export は `final.md` から生成し直すため、**export 後に手で足した定型ブロック（今回は「生成AIで作成」開示バナー）が消える**。付け直しが必要だった（→ 提案 H）。

## 3. 根本原因（なぜ公開物まで通り抜けたか）

- **R1: `reachable` が LLM 自己申告のまま信頼されている。** factchecker が `reachable:"ok"` と書いた値が、HTTP 未検証でも台帳・参考章に通る。maibun は実体 NXDOMAIN なのに `"ok"` で出荷された。「読んだ＝生きている」ではない。
- **R2: 到達性の機械確認（`article:sources-check`）が opt-in で、公開前の標準手順に入っていない。** jomon では一度も実行されず、CLAUDE.md・write-article でも必須化されていない。
- **R3: 仮に `sources-check` を回しても、現行の保守的マッピングと `references` の除外条件では取りこぼす。** 既定マッピングは **`dead` を 404/410 のみ**とし、NXDOMAIN・接続拒否・タイムアウト・5xx・403 は **すべて `unknown`**（誤検出回避の方針として妥当）。一方 `references`（[src/cli/references.ts](../src/cli/references.ts) `selectReferenceSources`）と `verify-artifacts` が弾くのは **`reachable:"dead"` のみ**。よって **「真に死んでいるが unknown 止まり」（＝ NXDOMAIN の maibun）は参考章に残る**。今回は手動で `dead` に格上げして初めて除外された。
- **R4: リンク腐敗（link rot）の時間依存性に対応していない。** 作成時に生きていても、公開後・時間経過で死ぬ。`checkedAt` の鮮度を見たり、公開済み記事を再確認したりするしくみがない。
- **（副次）R5: export 後付けの定型ブロックは再 export で失われる。** 公開メタ（AI開示等）を恒久化する経路がない。

要するに、**到達性の「正本」が機械検証になっておらず、機械確認も opt-in、確認しても unknown は素通り、しかも一度きり**——という4段の穴が重なった。

## 4. しくみ改善案

既存資産（到達性メタ／`sources-check`）を土台に、上の穴を塞ぐ。多くは独立に入れられるが、**C-1 は D に依存**（現行 `FetchOutcome` は `{status} | {error:string}` で error を全部 unknown に潰すため、NXDOMAIN と timeout を区別するには先に D の error 細分が要る）し、**A 軽量版・C-2・B は相互作用する**（後述）ので、導入順は §5 に従う。

### 提案A: `reachable` の出所を区別し、機械検証を信頼条件にする（R1）
- **現状**: `reachable` は factchecker（LLM）でも `sources-check`（HTTP）でも同じフィールドに書け、区別がない。
- **変更**: 出所を持たせる。`reachableSource: "llm" | "http"`（optional）を追加し、`checkedAt` と併せて「いつ・どう確認したか」を台帳に残す。**公開前ゲート（提案B）は `reachableSource:"http"` かつ十分新しい `checkedAt` の `"ok"` のみを信頼**し、LLM 申告の `"ok"` は「未検証」として扱う。
- **代替（軽量版）**: 出所フィールドを足さず、**factchecker は `"ok"` を書かない**運用に倒す。`"ok"` の確定は `sources-check`（HTTP）だけにする。スキーマ無改修で R1 を大幅に潰せる。
  - **重要（未記録と unknown の区別）**: スキーマは `reachable` の**省略（未記録）と `"unknown"`（確認したが不明）を意図的に区別**している（[src/schemas/ClaimsSchema.ts](../src/schemas/ClaimsSchema.ts) は default を付けない）。factchecker は **裏取り済みでも未到達確認なら `reachable` を省略**する（`"unknown"` を書かない）。これで「未確認の中立」が保たれ、C-2 の unknown 除外で draft 段階の参考章を壊さない（後述の C-2／受け入れ基準 #4 と連動）。`"unknown"` は機械（`sources-check`）専用の「確認したが断定不能」に限定する。`"dead"` のヒント記録は従来どおり可。
- **A 軽量版を採れば A 正式版は大半が不要**: 軽量版で「`"ok"` は `sources-check` だけが書ける」を担保すると **`"ok"` ⟹ http 検証済み**が構造的に成立するので、`reachableSource:"http"` を別に持つ価値は薄れる（出所区別が監査で要るときだけ残る）。両者は加算ではなく**ほぼ代替**。まず軽量版、フィールド追加は監査要件が出てから。
- スコープ外: 既存 run の retroactive fail（optional・後方互換を維持）。

### 提案B: 公開前の到達性ゲート（opt-in → 既定実施＋override）（R2）
- **現状**: `sources-check` は opt-in、`export` は到達性を見ない。
- **変更**: `export` で、**cited な source の到達性が「http 検証済み・dead なし・unknown は解決済み」であることを要求**。満たさなければ FAIL し、`--allow-unverified-links --note "<理由>"` でのみ通す（export 既存の「承認は `--note` で台帳に残す」流儀に合わせる）。
- **実装方式（確定）**: 判定は**純関数 `linkGate(claims, sources, opts): {pass, fails[], warnings[]}` を新設**し、`article:export` が既定で呼ぶ。CLI を分けず本体に内蔵し、必要になったら薄い `article:links-gate` を後付けする（純関数を共有）。**`verify-artifacts` は通信しない原則のまま**——ゲートも通信せず `reachable`/`checkedAt` の記録だけを見る（通信は `sources-check` に閉じる）。実質「公開前に `sources-check --only-cited` を回す」ことを強制する仕掛け。
- **`checkedAt` 欠落＝未検証＝FAIL（maibun の本質）**: maibun が抜けた真因は「`sources-check` が一度も走らず `checkedAt` が無かった」こと。ゲートは **cited に `checkedAt` が無いものを「未検証」として FAIL** させる（`reachable:"ok"` でも LLM 申告なら未検証扱い＝A と連動）。
- **鮮度閾値（確定）＝90 日。挙動はモード別に固定**（実装で割れないよう数値と分岐を確定する）:
  - **新規 export（`article:export`）**: `checkedAt` 欠落・90 日超・未解決 unknown・dead を **FAIL**（`--allow-unverified-links --note` で override 可）。
  - **公開済みの一括点検（提案E）**: 同条件を **warning 一覧**で出し、停止はしない（運用で順次対応）。
  - import/更新（`/update-article`）: 版を固定する import 時に cited の再取得（`sources-check`）を要求し、新規 export と同じ閾値で判定。
- **旧 run / offline の移行方針（retroactive に壊さない）**: §6 の「既存 run を retroactive に壊さない」と両立させる。**`checkedAt` を一度も持たない旧 run は、初回だけ warning に降格**（FAIL にしない）。一度でも `sources-check` を回した run（`checkedAt` あり）以降は通常どおり FAIL 判定。**ネット不通・CI 等で `sources-check` を回せない場合は `--allow-unverified-links --note "offline"` で明示 override**（無条件素通りはしない＝証跡を台帳に残す）。
- **除外・停止はこのゲート1か所に集約**（references 側は警告のみに留める。C-2 参照）。生成（references）と門番（gate）の責務を混ぜず、refine ループの安定性を壊さない。
- スコープ: 既定 ON だが override 可。CI 連携は別途。

### 提案C: 「確定的到達不能」を unknown から昇格させ、references でも警告/除外（R3）
- **現状**: NXDOMAIN・接続拒否は `unknown`。`references`/`verify-artifacts` は `dead` のみ除外。
- **前提**: マッピング細分化には D（error の細分＝NXDOMAIN／ECONNREFUSED／timeout の区別）が必要。現行 `classifyReachable` は error を全部 unknown に潰すため、**C-1 は D とセットで**入れる。
- **変更（二段）**:
  1. **マッピング細分化（NXDOMAIN を優先、ECONNREFUSED は据え置き）**: `sources-check` で **DNS 解決失敗（NXDOMAIN）はリトライ後も再現するなら `"dead"` 相当に昇格**（404/410 と同じ「恒久的に存在しない」クラス）。**ECONNREFUSED は `"dead"` に上げない**——メンテナンス・一時停止でも出るうえ、同一 run 内のリトライは数秒間隔でメンテ窓を跨げないため、偽 dead（生きている出典を消す逆方向の事故）のリスクが NXDOMAIN より高い。ECONNREFUSED・timeout・5xx・403 は `unknown` のままとし、**接続拒否は C-2 経路（公開前に可視化）／E の時間を置いた再確認で複数回 dead を確認できたものだけ昇格**を検討する。
     - **フラグ名の衝突回避**: `sources-check` 文書の `--strict-dead` は「5xx→dead」用に予約された名前。5xx（一時障害寄り）と NXDOMAIN（DNS 恒久失敗）は性質が違うので、**同じフラグに2つの意味を載せない**。昇格対象（5xx か DNS 失敗か）はフラグで明示的に分けるか別名にする。
  2. **references/ゲートの扱い（除外はゲートに集約）**: cited な `"unknown"` について、**`references` 自体は除外せず警告のみ**（draft の refine ループで参考章を壊さない）。**除外・停止の判断は提案B の公開前ゲート1か所に集約**し、「未解決 unknown あり」として FAIL させる。「迷ったら unknown」を作成時の安全弁として残しつつ、**公開直前は unknown を黙って素通りさせない**。
     - **ゼロ件 throw への配慮**: [src/cli/references.ts](../src/cli/references.ts) `prepareReferencesBlock` は選定0件で例外を投げる。C-2 を references 側で除外実装すると「全 cited が unknown の小記事」でパイプライン全体が FAIL する。除外を references でやらずゲートに寄せることで、この破壊を避ける。
- 効果: 今回 maibun（NXDOMAIN）が手動格上げなしで弾け、ECONNREFUSED の偽 dead は出さない。

### 提案D: プロービングの堅牢化で誤検出を減らす（R3 の副作用対策）
- **現状**: 単発リクエスト。今回 `rekihaku.city.yokohama`（timeout）/`web.joumon`（接続拒否）の切り分けに人手の再確認を要した。
- **変更**: HEAD→失敗時 GET フォールバック、ブラウザ相当 UA、限定的なリトライ。
- **リトライ仕様（確定・レイテンシ上限を切る）**:
  - **`2xx`／`404`／`410` は即確定・リトライしない**（確定的な結果を待たない）。
  - **リトライ対象は `timeout`／接続エラー（ECONNREFUSED 等）／DNS 一時失敗のみ**。指数バックオフ 2〜3 回。
  - **NXDOMAIN は短い再確認（1〜2 回）後に再現すれば `dead` 昇格**（ECONNREFUSED は昇格しない＝C-1）。
  - **1 URL あたりの合計待機上限 ~15 秒**（リトライ込み）。これを超えたら打ち切って `unknown`。公開前チェックのレイテンシ増を上限で封じる。
- **判定入力は typed に（C-1 のテスト可能性）**: 内部結果型を `errorKind: "nxdomain" | "timeout" | "connrefused" | "tls" | "other"`（status とは別）として**必須化**し、`classifyReachable` の入力を文字列パースでなく型で受ける。台帳保存（`httpStatus`・DNS 結果）は optional のままでよいが、**判定入力を typed にすることで NXDOMAIN→dead 昇格を fake で決定的にテストできる**。
- 効果: 生きているのに timeout/403 で落とす偽 dead を防ぎつつ、本当に死んでいるものを `dead` に上げられる。レイテンシは URL あたり上限で抑える。

### 提案E: 公開済み・シリーズ一括の再チェック（link rot）（R4）
- **現状**: 旧 run の一括再チェックはスコープ外（[課題-対策-実装計画-sources-check.md](課題-対策-実装計画-sources-check.md) §スコープ外）。
- **変更**: `article:sources-check` に **複数 run／シリーズ単位の一括モード**（例 `--series <slug>` や `--all-published`）を追加。**`checkedAt` が 90 日超（提案B と同一閾値）の cited source を「再確認推奨」として warning 一覧**で出す（一括点検は停止しない＝B のモード別挙動と一致）。定期実行（cron/CI）は任意の発展。
- 効果: 「作成時は生きていたが今は死んでいる」（今回まさにこれ）を定期的に拾える。

### 提案F: cited URL のアーカイブ保存（任意・強い対策）（R4）
- **変更**: 公開時に cited URL を Web アーカイブ（Wayback の Save Page Now 等）へ保存し、`archiveUrl`/`archivedAt` を台帳に記録。死リンク化しても参考章にアーカイブ版を併記できる。
- スコープ: 外部依存が増えるため opt-in。`sources-check` とは別 CLI。**best-effort（保存失敗・レート制限は warning に留め export をブロックしない）**——Save Page Now は失敗・制限が常態のため、ゲート（B）の必須条件にはしない。

### 提案G: 運用・テンプレートの改訂（R2 の即効策）
- [CLAUDE.md](../CLAUDE.md) 参考章の行に「**公開前に `article:sources-check --only-cited` を回し、dead/unknown を解決してから export**」を手順として明記（「任意」から「公開前の標準」へ格上げ）。
- [templates/.claude/agents/article-factchecker.md](../templates/.claude/agents/article-factchecker.md)：reachable を自己申告で `"ok"` にしない（提案A 軽量版）／到達確認は機械に委ねる旨を明記。
- [docs/qiita-article-howto.md](qiita-article-howto.md)：公開前チェックリストに死リンク機械ふるいを追加。
- write-article / update-article スキルの工程に pre-export の `sources-check` を組み込む。**ただし `sources-check` は外部通信のため allowlist には入れない方針は維持**し、スキルには「**公開前の標準工程だが外部通信なので明示承認を取って実行する**」と明記する（標準化と「通信は明示 opt-in」を両立させ、無断通信の運用事故を防ぐ）。

### 提案H（副次）: export 後付け定型ブロックの恒久化（R5）
- **現状**: AI 開示バナー等を export 物へ手追記しても再 export で消える。
- **変更**: `--front-matter` と同様に、**定型の告知ブロック（プラットフォーム別の「生成AIで作成」開示など）を export 時に注入するオプション**（`--notice <file>` / プロファイル設定）。本文（`final.md`）を汚さずに公開メタを焼き込める。
- スコープ: 死リンクとは独立。今回の運用で露見したため併記。

## 5. 優先度と段階導入

相互作用（A 軽量版・C-2・B、および C-1→D の依存）を踏まえ、副作用の小さい順に段階導入する。

- **第1段（即効・運用＋ごく小さい改修）**: 提案G（手順化）＋提案A 軽量版（factchecker は `"ok"` を書かず、**未確認は省略**・`unknown` は機械専用）。これだけで R1/R2 の大半が閉じ、refine ループへの副作用がない。
- **第2段（公開前ゲートに除外を集約）**: 提案B（公開前ゲート。`checkedAt` 欠落＝FAIL〔ただし旧 run は初回だけ warning に降格・§6 #8〕／90 日鮮度閾値／未解決 unknown を FAIL）＋提案C-2（**references は警告のみ・除外と停止はゲート B に集約**）。C-2 を references 側で除外実装してパイプラインを壊さないため、必ず B とセットで入れる。
- **第3段（プロービング堅牢化と dead 昇格）**: 提案D（error 細分・GET フォールバック・UA・リトライ）→ 提案C-1（**NXDOMAIN のみ dead 昇格、ECONNREFUSED は据え置き**）。D が前提。＋提案E（一括再チェック）。
- **任意（発展）**: 提案A 正式版（出所フィールド。A 軽量版でほぼ代替されるため監査要件が出てから）＋提案F（アーカイブ・best-effort）＋提案H（告知ブロック注入。死リンクと独立）。

## 6. 受け入れ基準（主要提案）

1. （G）公開前に `sources-check --only-cited` を回す手順が CLAUDE.md／スキル／factchecker テンプレに明記され、jomon と同型の見落としが手順上発生しない。
2. （A）factchecker 由来の未検証 `"ok"` が、公開前ゲートで「検証済み ok」と同等に扱われない（出所区別 or `"ok"` 非記録のいずれか）。factchecker は**未確認なら `reachable` を省略**し（`unknown` を書かない）、`unknown` は機械（`sources-check`）専用とする。
3. （C-1, D 前提）**NXDOMAIN がリトライ後も再現する場合 `dead` に分類**され、`references`/ゲートで除外される（maibun ケースが手動格上げなしで弾ける）。**ECONNREFUSED は `dead` に上げず**、偽 dead（生きている出典の除去）を出さない。
4. （C-2）cited な `unknown` は公開前に黙って素通りせず、可視化される。**除外・停止は公開前ゲート（B）に集約**し、`references` 自体は警告のみで参考章生成を FAIL させない（全 cited unknown の小記事でもパイプラインが落ちない）。
5. （B）`export` が未検証/未解決の cited リンク（`checkedAt` 欠落・**90 日超**・未解決 unknown・dead）を既定で止め、`--allow-unverified-links --note` 付き override でのみ通る。判定は**純関数 `linkGate` を `article:export` が呼ぶ**形で、通信せず記録から行う。鮮度閾値とモード別挙動（新規 export=FAIL／一括点検=warning／import=再取得）が固定されている。
6. （D）**`2xx`/`404`/`410` は即確定でリトライせず、リトライ対象は timeout/接続エラー/DNS 一時失敗のみ・1 URL 合計待機上限 ~15 秒**。判定入力は `errorKind`（nxdomain/timeout/connrefused/…）で typed 化され、NXDOMAIN→dead 昇格が fake で決定的にテストできる。`unknown` の内訳（DNS 解決可否・TCP 拒否・timeout）は台帳に optional 記録。
7. （E）シリーズ／公開済み run を一括で再確認でき、`checkedAt` が 90 日超（B と同一閾値）の cited source を warning 一覧で出せる（一括点検は停止しない）。
8. （移行）`checkedAt` を一度も持たない**旧 run は初回だけ warning に降格**し retroactive に FAIL させない。ネット不通/CI で `sources-check` を回せない場合は `--allow-unverified-links --note "offline"` で明示 override（無条件素通りはしない）。
9. いずれも既存 run を retroactive に壊さない（メタは optional・後方互換）。`npm test` 緑・typecheck クリーン。通信は `sources-check` に閉じ、`verify-artifacts` は通信しない原則を維持。
