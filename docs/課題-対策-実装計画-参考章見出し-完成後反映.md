# 課題・対策・実装計画：完成記事への参考章見出しの反映（初回確定＋自己修復整列）

> 前提: 参考章見出しの記事ごとカスタム（既定 `参考`・run 単位 first-write-wins）は実装済み（v0.2.57）。設計は [docs/課題-対策-実装計画-参考章見出しカスタム.md](課題-対策-実装計画-参考章見出しカスタム.md)。本書はその続きで、**完成記事（マーカーあり）で見出しがしくみで反映されない穴**を埋める。
>
> **スコープ（重要）**: 本書の主目的は「**`meta.referencesHeading` 未確定（既定 `参考`）の完成記事に、初回 heading を本文へ反映する**」＋「**確定済み heading に本文見出しを冪等に揃える（自己修復整列）**」の2点。**既に `参考` 以外で確定済みの heading を別文言へ変える**のは first-write-wins により `--heading` が無視されるため、本改修だけでは扱えない。それは §4 の再設定 override の領域（別 PR）。

## 1. 背景・課題

`article:references --heading "<text>"` を **完成記事**に対して実行しても、本文の見出し行（例 `## 参考`）が据え置きになり、手で `## <text>` に直す必要がある。

### 1.1 根本原因（コードで確認）

参考ブロックを本文へ反映する [replaceMarkedBlock](../src/cli/references.ts#L144) は2経路に分かれる。

| 状態 | 経路 | 見出し行の扱い |
|---|---|---|
| マーカー無し（新規作成・改名） | Case 2/3 | 設定見出しで新規作成、または旧 `## 参考` を adoption/rename する |
| マーカーあり（完成後の通常状態） | **Case 1** | [references.ts:154-163](../src/cli/references.ts#L154-L163)：`<!-- sources:begin/end -->` 間のブロックだけ置換して return。**`heading` 引数を使わない＝見出し行は据え置き** |

完成記事は当然マーカーありなので Case 1 を通り、`--heading` が `meta.referencesHeading` を更新しても**本文見出しは変わらない**。これが手動編集の原因。

### 1.2 どこまでが「しくみで完結」しているか

`meta.referencesHeading` の確定状態と本文のマーカー有無で整理する。

| ケース | 現状 | 本改修後 |
|---|---|---|
| 初回指定（create/series:init、または新規 references で本文マーカー無し） | 完結（Case 2/3 が設定見出しを使う） | 同左 |
| **meta 未確定**（既定 `参考`）の**完成記事**に初回 `--heading` | Case 1 を通り本文見出しは据え置き＝**手当て要** | **§3 整列で完結**（meta 初回確定＋本文 rename） |
| meta 確定済みで本文見出しが旧 `## 参考` 等のまま（過去 run の名残） | 据え置き | **§3 整列で自己修復**（`--heading` 不要） |
| **meta が `参考` 以外で確定済み**の heading を**別文言へ変更** | `--heading` は first-write-wins で無視 | **本改修では不可**。§4 の override（別 PR） |

→ 本書が「機能改修で埋める」のは中段2行（meta 未確定の完成記事への初回反映＋確定済みへの自己修復整列）。最下段（確定済みの文言変更）は §4 に切り出す。

## 2. 方針

**Case 1（マーカーあり）でも、マーカー直前の参考章見出し行を「解決済み見出し（`meta.referencesHeading`）」へ揃える（冪等な自己修復）。** これでブロックの有無に依らず見出しまで反映され、手動編集が不要になる。検証ゲート（verify-artifacts / stats / series:check）は**無改修**（正本はマーカーで見出しはブロック外＝見出し非依存）。

設計判断は §6 の評価サマリに従い、以下2点を提案へ補正する。

- **A. `--heading` の有無に関わらず常に解決済み見出しへ整列する**（冪等の自己修復）。整列対象は引数 `heading`（＝CLI/meta から解決済み）であり、`--heading` 指定時に限らない。これにより「create 時に見出しを焼いたが当時の references が Case 1 を通って本文は `## 参考` のまま」という既存 run も、次の `article:references`（`--heading` 無し）で自動的に揃う。
- **B. rename 対象を「安全側」に限定する**。マーカー直前（fence 外・空行はスキップ）の非空行が**既知の参考見出し**に一致する時だけ rename。一致しなければ no-op（直前が通常本文・散文のケースを誤改名しない）。

## 3. 仕様

### 3.1 Case 1 の新しい挙動（マーカーあり）

`replaceMarkedBlock(body, begin, end, block, heading)` の Case 1 を次のとおり拡張する。

1. 従来どおりマーカー区間 `[begin, end]` を `block` で置換する。
2. **加えて**、`begin` マーカーの直前にある参考章見出し行を探し、`## <heading>` と異なれば `## <heading>` に rename する（冪等）。
   - 探索: `body.slice(0, bIdx)` を prefix とし、その末尾の行から上へ走査。**fence 外**かつ**空行はスキップ**し、最初に当たった非空行を見る。
   - rename 条件: その行が **rename 対象見出し集合**（§3.2）に一致し、かつ既に `## <heading>` でないこと。
   - 一致しなければ no-op（誤改名しない）。マーカーが行頭でない異常配置も prefix 末尾が見出しにならず no-op＝安全。
3. 返り値の `status` は `replaced` を維持。rename が起きたかは `warnings`（任意）かログで可視化する（§3.4）。

### 3.2 rename 対象見出し集合（安全な matcher）

**rename 元（どの見出し行を整列対象として拾うか）と rename 先（何に揃えるか）を分けて定義する。**

**rename 先**: 常に `## <heading>`。`heading` は `validateReferencesHeading` を通った解決済み見出しのみ（§3.1 の引数）。**LLM 系の語（`参考資料`/`出典` 等）は validation で保存不可なので、rename 先になることはない**。

**rename 元**（マーカー直下のこの集合に一致する行だけ拾う）:

- 既定 `参考`（`REFERENCES_HEADING_RE` = `/^##\s+参考\s*$/`）。
- 旧 LLM 系の参考見出し（`参考リンク`/`参考文献`/`参考資料`/`参考URL`/`出典`/`参照`/`References` 等＝`LLM_REFERENCE_HEADING_RE` 相当）。完成記事のマーカー直上にこれらが残っているのは過去経路の名残なので、設定見出しへ寄せてよい（＝**旧 LLM 系は rename 元としてのみ許す**）。
- 現設定見出し `headingMatcher(heading)`（既に一致＝no-op の判定に使う。`heading` は validation 済みなのでここに LLM 系は来ない）。

> 補足: `heading` は validation 済みなので「現設定見出しが LLM 系」という no-op ケースは構造上起きない。`## 出典` 等を拾うのは「マーカー直前が旧 LLM 系なら rename 元にする」意味であり、矛盾しない。
>
> マーカー直下の参考見出しは「機械生成の参考章」とみなすのが妥当なので、この集合に限れば通常本文の誤改名はほぼ起きない。

### 3.3 冪等性

- 整列先（`## <heading>`）と現見出しが一致するなら、`replaceMarkedBlock` は見出しを書き換えない。
- ただし `article:references` は**現状ブロック置換の有無に関わらず常に** `final.references.bak.md` と `final.md` を保存する。整列が no-op でも `content === final`（ブロック内容も完全一致）になるとは限らない（再 normalize で `retrievedAt` 等が変わりうる）。**「見出し差分が出ない」ことが本書の冪等保証**であり、保存自体の skip は別途 §5 の任意ステップで `content === final` のときだけ skip する場合に成立する。
- 同じ run で `article:references` を繰り返しても、**見出しは設定見出しに収束し変化しない**（これが満たすべき不変条件）。

### 3.4 可視化

- rename が起きたら CLI（`article:references`）で stderr に一行出す（例: `参考章見出しを「参考」→「参考・題材として確認した公式情報」に整列しました`）。
- `replaceMarkedBlock` は純関数を保ち、rename 情報は `ReplaceResult.warnings`（既存）に積んで**呼び出し側が出力**する（§責務境界は既存方針を踏襲）。

### 3.5 「meta が正、本文は派生」の明確化

ユーザーが本文見出しを設定と**違う**文言に手で変えていた場合、整列で設定見出しへ戻す。これは「`meta.referencesHeading` が正・本文見出しは派生」という方針として一貫させる。意図的に別文言へ変えたいときは §4 の再設定 override で meta 側を更新する運用に誘導する。

## 4. 任意：再設定（heading の変更）の override

**主目的（完成後に1回設定）には不要**。その時点で `meta.referencesHeading` は未設定（既定 `参考`）なので first-write-wins が初回書き込みを許し、§3 の整列だけで完結する。

「一度決めた見出しを後で別文言に変える」だけは first-write-wins が壁になる。判断材料と推奨：

- 見出しは表示ラベルで検証に無関係（正本はマーカー）。editor-model のような監査値とは性格が違う。
- 無条件 last-write-wins は「再 references で別 `--heading` を渡して気付かず変わる」事故余地を残す。

→ **推奨: first-write-wins を維持しつつ、意図的変更だけ明示フラグで許す**（例 `article:references --reheading "<text>"`、または `--heading` ＋ `--force-heading`）。override 時は旧値・理由を stderr かイベントに残す（「状態変更は明示・台帳化」の流儀）。**これは主改修と分離した第2弾**とし、本スコープから外してよい（同梱可否は実装時に判断）。

## 5. 実装計画

### 第1段（主改修・最小）

1. **references.ts**
   - prefix 末尾の参考見出し行を探して rename するヘルパ `alignTrailingRefHeading(prefix, heading): { prefix: string; renamedFrom?: string }` を追加（§3.1-3.2、`seriesCheck.stripReferenceBlock` の scan-back と同型・fence 判定は prefix 末尾の単純走査で足りる）。
   - 既知参考見出しの matcher を共通化（`REFERENCES_HEADING_RE` ＋ `LLM_REFERENCE_HEADING_RE` ＋ `headingMatcher(heading)`）。
   - `replaceMarkedBlock` の Case 1 を「ブロック置換＋ prefix 整列」に変更。rename が起きたら `warnings` に積む。`status` は `replaced` 維持。
2. **index.ts `article:references`**
   - 既存どおり `warnings` を stderr 出力（replaceMarkedBlock の warnings 経路に rename 通知が乗る）。追加配線は不要〜最小。
   - **（任意）no-op 保存 skip**: `content === final` のとき `final.references.bak.md` / `final.md` の保存を skip して「unchanged」を出力する。これを入れて初めて「整列済み再実行で bak を作らない」が成立する。入れない場合は §3.3 のとおり保証は「見出し差分が出ない」に留める（本文 byte 完全一致は再 normalize で崩れうるため過剰約束しない）。
3. **テスト**（§7）。

### 第2段（任意・再設定 override）

4. `article:references --reheading <text>`（または `--force-heading`）で first-write-wins を意図的に上書き。meta 更新＋§3 整列＋旧値ログ。要否はユーザー判断。

## 6. 影響しない箇所（無改修で確認）

- **verify-artifacts**：マーカー基準。見出し rename は無影響。
- **stripLlmReferenceSections**：設定見出しを除外済み。整列後 `## <設定見出し>` は LLM 節と誤認されない。
- **series:check**：マーカー優先＋直前見出し行除去済み（v0.2.57）。整列でむしろ一貫。
- **stats**：`SOURCES_BEGIN` 基準。無影響。

## 7. テスト計画

- Case 1：マーカーあり＋本文 `## 参考`、heading=「X」→ 見出しが `## X` に整列・ブロック更新・二重化なし。
- Case 1：`--heading` 無し（meta 由来 heading）でも本文見出しが meta の見出しへ自己修復。
- Case 1：マーカー直前が通常本文（参考見出しでない）→ **無改名**（誤改名しない）。
- Case 1：旧 LLM 系見出し（`## 出典` 等）がマーカー直上に残存 → 設定見出しへ整列。
- 冪等（見出し）：整列済みに再度 `replaceMarkedBlock` → **見出し差分は出ない**（`status=replaced`・rename warning 無し）。
- 冪等（保存 skip・任意ステップを入れる場合のみ）：`content === final` で `article:references` が保存を skip し「unchanged」を出す。
- 回帰：既存 Case 2a/2b/2c・マーカー破損 throw は不変。

## 8. リスクと対策

| リスク | 対策 |
|---|---|
| 通常章の誤改名 | rename 対象を「既知参考見出し＋マーカー直下」に限定（§3.2）。一致しなければ no-op |
| 手編集の見出しを意図に反して戻す | 「meta が正・本文は派生」を明確化（§3.5）。別文言にしたいなら §4 override で meta を更新 |
| 冪等性（見出しが毎回揺れる） | 整列先＝現見出しなら見出しを書き換えない＝**見出し差分は出さない**（§3.3）。bak 生成の抑制は保証外（任意の `content === final` 保存 skip を入れた場合のみ。§5-2） |
| マーカー破損時の事故 | 従来どおり wellFormed 後にのみ整列（破損は先に throw） |

## 9. 非対象

- マーカーブロック内部の描画フォーマット（`renderReferencesBlock`）。
- 参考リンクの選定・並び順。
- 見出しの多言語化・テンプレート変数。
- 検証ゲートのキー（マーカー）変更。
