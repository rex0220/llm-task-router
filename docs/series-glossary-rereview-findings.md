## 1. 確認できたこと（第1段は機能している）

- **glossary 定義語はゼロ問題**。`竪穴建物/竪穴住居` は全記事で「竪穴建物」を本語に統一し、初出のみ「（…かつて竪穴住居とも呼ばれた…）」と併記＝`firstUseAlias: per-article` に完全準拠。`三内丸山遺跡` の所在地も本文はすべて「青森市」。
- つまり **`series:check` は「人が正本に決めた語にシリーズが揃っているか」を確実に担保できている**。設計の中核（正本＝glossary・機械照合・read-only）が実データで効いた。

## 2. 残った2系統の課題

### 課題A：既知ノイズが実データで再確認された（参考章のソース正式名）

`series-check-report` の青森県 2件は、いずれも**機械生成の参考章のソース固有名**:

- intro: `[S006] …（青森県公式）`
- people: `[S008] …（青森県・特別史跡三内丸山遺跡 公式サイト）`

これは出典サイトの正式名称であって**本文の表記ゆれではない**＝誤検知。重要なのは、**今回の検出ノイズ 2/2 件すべてが参考章由来**だったこと。提案書 §4 で「将来候補」としていた**参考ブロック除外**を前倒しする実データ根拠ができた。

### 課題B：glossary 未定義の実ゆれは第1段では拾えない（設計どおりの空白）

人手レビューで初めて見つかった2件は、いずれも glossary に `preferred`/`variants` を定義していない語の揺れ。`series:check` は定義語しか見ないので**検出しない（仕様どおり＝漏れではなく対象外）**。

- **① 堅果（木の実）名のカナ/かなゆれ（最優先）**: シリーズ標準はカタカナ（life:29 ＝ ドングリ・クリ・トチ・クルミ）。対して intro:75 は「どんぐりやくりのような木の実…」とひらがなで書き、**同一文の後半で「クリ」とカタカナ**＝1文内で割れ、かつシリーズ標準とも不一致。
- **② 人びと／人々**: シリーズ標準は「人びと」（ひらがな・全記事一貫）。people:82（見出し「日本列島の人々」）と people:211（「アイヌや琉球の人々」）の2か所だけ「人々」（漢字）＝jomon-people のみのドリフト。

この2件は、**「未知の揺れを辞書なしで見つける」能力が第1段に無い**ことの実証。glossary に無い以上、機械は気づけない。

### 非対象（ゆれ扱いしない）

- **ムラ／集落**: 全記事で混在するが「ムラ＝親しみ表現／集落＝学術語」の使い分けに見え、`voice.md` の方針とも整合＝**意図的**。glossary に入れると過検出になるため**登録しない**。

## 3. 対応案

### A. 参考ブロック除外（既知ノイズの根絶・小粒・最優先）

`series:check` の照合対象から、機械生成の参考章を除外する。

- **除外範囲（停止条件を明記）**: 段落走査の前に、次の優先順で参考ブロックを本文から落としてから `splitParagraphs` に渡す。
  1. **第一優先＝マーカー**: `<!-- sources:begin -->` … `<!-- sources:end -->` の区間を除外（`article:references` が必ず撒くので最も確実・閉じ忘れが無い）。
  2. **fallback＝見出し**: マーカーが無い場合のみ、`## 参考` 見出し行から「**次の同階層以上（`#`〜`##`）の見出し、または EOF まで**」を除外。次の `###` 以下は参考章の内側とみなし含めて落とす。
  - 対象見出し文字列は `## 参考` を既定とし、`## 参考リンク` / `## 出典` も同義として落とす（提案書で機械生成の `## 参考` に一本化済みだが、過去 run の揺れに備えて別名も拾う）。見出しの前後空白はトリムして一致判定。
- **効果**: 今回の誤検知 2/2 件が消える。**glossary 側の変更は不要**。
- **整合**: 参考章の正しさ（台帳一致）は `verify-artifacts` が別途担保する。`series:check` は本文に集中、という責務分離を保つ。
- **位置づけ**: 提案書 §6 の後段「参考ブロック除外」を**第1.5段（小粒）に前倒し**。

### B. glossary 未定義ゆれへの対応（2路線）

①②は性質が異なる。短期は B1（即効）、本命は B2（発見の自動化）。

気づいた揺れを辞書化して、以後は機械が守る。ただし**短語の部分一致リスクで2段に分ける**（B1a は現スキーマで安全、B1b はスキーマ拡張が前提）。共通の長所/短所:

- **長所**: 気づいた揺れを今すぐ機械が守れる。発見ループ（B2→B1）の受け皿。
- **短所**: 「正本を知っている揺れ」しか拾えない＝**事前に気づいた分だけ**。①②は人手レビューで初めて見つかった＝辞書化はつねに後追いで、網羅性に限界がある（だから本命は B2）。

#### B1a. 人びと／人々 だけ先に拡充する（現スキーマのまま・安全）

「人びと」「人々」はどちらも他語の一部に出にくい**自立性の高い語**なので、現行の単純部分一致でも誤検出が起きにくい。現スキーマのまま登録できる。

```yaml
terms:
  - preferred: 人びと
    variants: [人々]
    firstUseAlias: false   # 常に統一（初出例外なし）
```

#### B1b. 堅果名は terms スキーマ拡張の後（短語の部分一致対策が前提）

`クリ`/`くり` のような短語は他語の一部に出る（例: 「びっ**くり**」に「くり」、「**クリ**ック」に「クリ」）。**現スキーマの単純部分一致のまま堅果名を登録すると確実に誤検出する**ので、`terms` に次のいずれかを足してから入れる:

- `wordBoundary: true`（語境界での一致のみ。日本語は空白境界が無いので「前後がカナ連続でない」等の近似境界を定義する必要あり＝要設計）、または
- `matchMode: "exact-token"`（B2a の語抽出＝カナ連続列トークンに対する完全一致。下記 B2a の抽出単位と共有できる）、または
- `context`（堅果の文脈語 `木の実`/`堅果`/`採集`/`食` 等が同一段落にあるときだけ検出。nouns の `contextPatterns` と同じ仕組みを terms にも持たせる）。

```yaml
# スキーマ拡張後のイメージ（堅果名）
terms:
  - preferred: クリ
    variants: [くり]
    firstUseAlias: false
    matchMode: exact-token     # or wordBoundary: true / context: [木の実, 堅果, 採集]
```

> **判断**: B1 を「設計変更なし」で一括投入しない。**人びと（B1a）は即・堅果名（B1b）は terms の語境界/トークン/context 対応を小さく入れてから**。順序を逆にすると堅果名の誤検出で `series:check` の信頼（false-positive 最小の方針）が崩れる。

#### B2. 辞書なしの「未知ゆれ」検出（本命・段階導入）

人手でしか見つからなかった①②を、機械が**先に候補として挙げる**仕組み。2方式。

- **B2a. 正規化クラスタリング（機械・LLM 不要）**: 本文中の語を正規化キー（カタカナ⇄ひらがなの kana-fold 等）でグルーピングし、「**同じ正規化キーなのに表記が2種以上**」を揺れ候補として挙げる。
  - **候補語の抽出単位（最小仕様）**: 日本語は空白で区切れないので「何を1語と切るか」を先に決める。第2.5段の最小実装は次:
    1. **カナ連続列を1トークンとして切り出す**（`[ぁ-んァ-ヶー]+` の最長一致。漢字・英数・記号で区切れる）。漢字交じり語は対象外（B2a の射程はカナ表記ゆれ）。
    2. **長さ2文字以上**のみ候補（1文字は助詞・語尾でノイズが多い）。
    3. **stoplist で一般語・助詞相当を除外**（例: 「こと」「もの」「ため」「よう」「など」「これ」「それ」、長音・促音だけの断片等）。
    4. 残ったトークンを kana-fold（カタカナ⇄ひらがなを同一視）した正規化キーでグルーピングし、**1キーに表記が2種以上**なら候補として記事・段落・出現箇所つきで出す。
  - これで①（くり↔クリ・どんぐり↔ドングリ）は**辞書なしで高精度に拾える**。同一文内・シリーズ標準逸脱の発見に最適。抽出したトークンは B1b の `matchMode: exact-token` の一致単位とも共有できる。
  - ②の漢字/かな（人びと↔人々）は読みが同じでも正規化に形態素解析が要り、カナ連続列抽出では拾えない＝**B1a（対応表で持つ）側の担当**。B2a の射程はカナ表記ゆれに限る、と明示する。
- **B2b. LLM 補助（提案書 §4 第2段の本丸）**: 段落／記事を LLM に渡し「シリーズ標準から外れた表記ゆれ候補」を挙げさせ、人が確認して glossary に昇格。コスト高・要レビューだが、辞書にも正規化にも乗らない揺れを拾える。
- **役割分担**: B2 が「候補発見」、B1/glossary が「正本固定」。**発見（B2）→ 人が正本決定 → glossary 登録（B1）→ 以後 `series:check` が機械的に守る**、という運用ループにする。

### 検証結果（A＋B1a を当てた実走）

jomon 6本に対し、A（参考ブロック除外）を実装し B1a（人びと/人々）を glossary に追加して再走した結果:

- **`totalFindings`: 11 → 2**。
- **A の効果（参考章ノイズ根絶）**: intro の `[S006]（青森県公式）`（final.md 152行目＝参考章）と people の `[S008]`（260行目＝参考章）の青森県2件が**除外**。本文に青森県の三内丸山文脈は残っておらず（people 本文は「三内丸山遺跡（…青森市にある…）」に是正済み）、noun findings は **0**。
- **B1a の効果（人びと/人々 検出）**: people の 2件（見出し「## …日本列島の人々…」と Q&A 行「**Q. アイヌや琉球の人々…**」）を**新規検出**。残った2件はこれ＝いずれも実害のあるドリフト。
- **竪穴建物/竪穴住居**: 記事側が「竪穴建物」本語＋初出のみ併記に是正済みのため、`firstUseAlias: per-article` 例外で **0 件**（誤検出なし）。
- 結論: **残検出はすべて真陽性（人々2件）・既知ノイズ0・誤検出0**。第1.5段 A で誤検知の出どころ（参考章）を根絶でき、B1a で未定義だった人びとゆれを正本化＝機械が守れるようになった。

## 5. 記事側の扱い（参考・今回は実施しない）

今回ユーザー判断は「**検証のみ・修正なし**」。①②を将来直す場合は CLAUDE.md 原則どおり:

- `final.md` は直接編集せず `article:revise --instruction-file` 経由で戻す。
- 本文変更後は `article:factcheck-scope` を回し、非事実差分（表記の統一）なら `--accepted-after non-factual-diff` で stamp。
- 直す際の正: ① カタカナ（くり→クリ・どんぐり→ドングリ）、② 人びと（人々→人びと）。

## 6. まとめ

- **第1段の中核は実証済み**: 「正本に揃っているか」を定義語ゼロ問題で守れている。
- **残るは2系統**: (A) 既知ノイズ＝参考ブロック除外で根絶（小粒・最優先）、(B) 未定義ゆれ＝**発見（B2）→正本化（B1）のループ**で段階対応。
- **B1 は一括投入しない**: 人びと（B1a）は自立語で現スキーマのまま安全だが、堅果名（B1b）は短語の部分一致で誤検出するため、`terms` の語境界/トークン/context 対応（B2a の抽出基盤と共有）を入れてから。順序を逆にすると false-positive 最小の方針が崩れる。
- 「**拾わないものを明示して段階導入する**」という第1段の方針の正しさを、再レビューが裏づけた（誤検知の出どころが1か所＝参考章に収束、未定義ゆれは設計どおりの空白）。
