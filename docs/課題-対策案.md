# 記事作成の課題と対策案

`docs/課題.md` で挙がった課題への対策案。既存実装（`runs/router.log`・stderr 進捗・`utils/cost.ts` のローカル概算・`.claude/settings.json` の allowlist）を土台に、小さく足せるものから並べる。

---

## 1. 進捗状況がわからない（開始・終了・処理時間をログで把握したい）

**現状**: stderr に `[n/total] name (task) ... → done via provider/model (elapsedMs, ~$cost)` が出る。ただし stderr は流れて消え、工程をまたいだ全体像が残らない。`runs/router.log` はモデルコール単位の JSONL（task / elapsed_ms / tokens / cost_usd）で、工程名や開始/終了は持たない。

**対策案**:
- **A. 正本は `progress.json`（機械可読）、`progress.md` は表示用に再生成**（推奨・重要）。`meta.json.steps` は既存互換/補助の位置づけ（正本は `progress.json` に一本化）。
  - ⚠ Markdown 追記だけを正本にすると、再実行・`resume`・途中失敗で行が**重複**しやすい。`progress.json` に「工程名・状態・startedAt・finishedAt・elapsedMs・概算コスト・出力パス」を upsert（同一工程は上書き）し、`progress.md` はそこから**毎回再生成**する。
  - 列例: `| # | 工程 | 状態 | 開始 | 終了 | 所要 | 概算$ | 根拠/出力 |`
- **B. 7工程すべてを記録するには「誰が書くか」を決める**（重要）。
  - 現在の `WorkflowEvent` は `create` / `evaluate` / `revise` 系にしか載らず、**factcheck / build-verify / claims-normalize / verify-artifacts / publication-check までは自動で覆えない**。
  - ⚠ **サブエージェントに `progress.json` を直接書かせない**（同時実行で競合・JSON 破損が起きやすい）。代わりに次のどちらか:
    - **(推奨) `progress.events.jsonl` への追記 + 再生成**: 各工程は「イベント1行」を append-only の `progress.events.jsonl` に追記するだけ（追記は壊れにくい）。`progress.json` / `progress.md` は最後にイベント列から**再生成**する。
    - **`llm-task-router progress:event` CLI ヘルパ**を1本用意し、CLI もサブエージェントも**必ずこのコマンド経由**で記録する（書き込みロジックを1箇所に集約。直接 JSON を触らせない）。
  - CLI コマンド（claims-normalize / verify-artifacts など）は終了時にこのヘルパ/追記でイベントを残す。CLI とサブエージェントで**同じイベントフォーマット**を共有する（編集長がオーケストレーション）。
- **C. `article:status --run <id>` を追加**し、現在地（課題の「7工程中6番目」表）を表示。再開判断にも使う。
  - 実装順を1つに決める: **`progress.events.jsonl`（append-only）を正、`progress.json`/`progress.md` は派生**とし、`article:status` は実行時に events が `progress.json` より新しければ**再生成してから読む**（=「読む直前に再生成」）。各イベント後の即時再生成は任意の最適化。
- **D. stderr 出力に絶対時刻 `[HH:MM:SS]` を付与**（現状は elapsedMs のみ）。tail で追える。

## 2. ファクトチェックに時間がかかる / 同じ検証の二度手間

**現状**: 編集レビュー後の本文変更に追従するため「再 factcheck → 正規化」を回す運用。前提が変わっていなければ二度手間。

**対策案**（実装難度はやや高め。`final.md` 全体ハッシュ「だけ」では不十分な点に注意）:
- ⚠ 全文ハッシュだけだと「**文体だけ変わった**」「**見出しの移動**」「**同じ主張の言い換え**」を誤検出/見逃す。snapshot・claim 単位・節単位を**組み合わせる**設計にする。
- **A. factcheck 時点の snapshot を残す**（推奨の土台）。factcheck 実行時の `final.md` を `runs/<id>/factcheck.snapshot.md` として保存し、再検証時はこれと現 `final.md` の差分を取る。
- **B. 差分は claim hash ＋ changed sections で判定**: `claims.json` の各 claim が指す本文箇所の hash と、`article:update-diff` の changed-sections を突き合わせ、「事実に関わる節/主張が変わったか」を判定。事実差分ゼロなら **再 factcheck をスキップ**し progress に `skipped (no factual diff)` を記録。
  - ⚠ **`claims.json` がまだ無い段階**を前提にしない。課題の実例（6番目「再 factcheck 中で、正規化済み `claims.json`/`sources.json` はまだ無い」）のように、normalize 前に再検証が走るケースがある。対策は2択:
    - **(運用)** 初回 factcheck の直後に必ず `claims-normalize` を回し、以降は `claims.json` が在る前提にする。
    - **(設計)** `claims.json` が無くても、`claims.raw.json` ＋ `factcheck.snapshot.md` の差分だけで判定できる経路を用意する（id 採番前でも raw の主張テキストと snapshot で足りる）。
- **C. 検証スコープを2段にする**: 初回=全文、2回目以降=変更のあった節・claim のみを factchecker に渡す（update フローで実績ある方式を新規作成にも展開）。verified 済み claim は当該本文が無変更なら verified を引き継ぐ（再Web検索しない）。

## 3. 時間がかかりすぎる（方向性チェックを前倒し・所要見積もり）

**対策案**:
- **A. 方向性ゲートを factcheck の前に置く**（推奨）。高コストな factcheck/build の前に、編集長が「方向性 OK/要修正」を `runs/<id>/direction-check.md` に記録する軽量ゲート。
  - 読む対象は工程で分ける: **create 直後なら `draft.md`**、**refine 後なら主成果物の `final.md`**（refine の主成果物は `final.md` であり `draft.md` ではない）。
- **B. 所要時間の見積もり表示**: 見積もりに使う実績の出どころに注意。
  - ⚠ factcheck / build-verify は**サブエージェント側の作業時間が主**で、`router.log`（OpenAI/Anthropic の API コール分）には現れない。`router.log` 平均で見積もれるのは create/evaluate/revise 系のみ。
  - 対策: **`progress.json` に各工程の実績 elapsed が溜まってから**、それを使って「factcheck 目安 約N分」を出す（実績ベース）。`router.log` 由来の見積もりは API 工程に限定する。
- **C. 重い工程（factcheck / build-verify）の並行化**: 互いに独立なので、編集長が両サブエージェントを同時起動して待ち時間を短縮。

## 4. API 費用をログに出したい（処理毎＋合計）

**現状**: stderr に工程毎 `~$x` と末尾 `total: ~$x (estimate)` が既に出る。`router.log` にも `cost_usd` がある。残課題は「残らない・記事単位で合算されていない」点。

**対策案**:
- **A. コストの正本は `progress.json` に寄せる**（推奨・重複管理を避ける）。`cost.json` を別に持つと正本が二重化する。まずは `progress.json` の `steps[].costUsd` と `totalCostUsd` で足りる。完成報告にはそこから貼る。
  - 別ファイルが必要になった場合のみ `cost.json` を `progress.json` から**派生（再生成）**する位置づけにし、正本にはしない。
- **B. `models.yaml` の `prices` 鮮度に注意**: 概算は価格表依存。古いと金額がズレるので、「使用した価格表の日付」を併記し、**「概算」である旨を明示**する。

## 5. Claude Code の使用量をログに残したい

**注意**: Claude Code（外側オペレーター/サブエージェント）のトークン使用量は llm-task-router の `router.log`（OpenAI/Anthropic API 呼び出し分）には**含まれない**。別系統。

**対策案**:
- **A. 工程単位の手動記録**: factcheck / build-verify / editorial の各サブエージェントが、終了時に自分の所要時間と「概算規模（読み込み行数・実行コマンド数）」を `runs/<id>/agent-usage.md` に1行追記する運用ルールにする（完全な課金額は取れないため目安）。
- **B. 正確な使用量が必要なら** Claude Code 側の利用状況（`/cost` 等のセッション情報）を完成報告に手で転記する。ツールでの自動取得は本リポジトリ外なので運用でカバー。

## 6. コマンド実行承認が多すぎる（記事フォルダー内は自動承認に）

**現状（確認済み）**: `templates/.claude/settings.json` の allowlist には pipeline 系が**既に広く入っている** — create / refine / evaluate / revise / resume / review / import / **update-diff / claims-normalize / claims-recheck / verify-artifacts / review-editorial**。`export` と build-verifier の `npm install`/`tsc`/`node` は意図的にプロンプト残し。つまり「コマンドを足す」より、**allowlist にマッチしない叩き方**が承認過多の主因。

**対策案**:
- **A. allowlist にマッチしない呼び方を運用監査で潰す**（推奨）。`cd ../foo && llm-task-router ...`（`cd &&` 連結はパターン不一致になりがち）や `npx llm-task-router ...`、相対パス起動などは allow に当たらず毎回プロンプトになる。**コマンドを足すのではなく、叩き方を allowlist に合う形に統一**する（カレントディレクトリ基準で実行・`npx` を避ける等）。実際にプロンプトが出た履歴を棚卸しして原因パターンを特定。
- **B. 抜けている許可だけ最小追加**: 監査で「正当だが未許可」のパターンが見つかった場合に限り `permissions.allow` に追記（むやみに広げない）。
- **C. `export` / `record-publication` / 未知コード実行（build-verify）はプロンプトを維持**（公開相当・任意コード実行は中身確認が必要なため、自動承認しない方針は堅持）。

---

## 7. 完成報告もログに残したい

**現状**: 完成報告はチャットに出すのみで `runs/` に残らない。

**対策案**:
- **A. 完成報告を `runs/<runId>/completion-report.md` に閉じて保存**（推奨）。課題.md の「完成報告の例」をテンプレ化し、ゲート結果表・構成・概算コスト・GO/NO-GO・上申事項を機械生成 or 編集長が確定して書き出す。
- ⚠ **`export/index.json` には混ぜない**。`export/index.json` は `record-publication` 側の**公開台帳**（公開 URL・版の正本）。ここに完成サマリを足すと「公開記録」と「ローカル書き出し/完成報告」が混ざる。**完成報告は `runs/<runId>/` に閉じる**のがきれい。
- **B. progress.json（課題1・費用も内包）/ progress.md（再生成）/ completion-report.md（本項）を1セット**にすれば、1記事の「進捗・費用・完成」が `runs/<id>/` 内で完結する。

## 優先度（codex レビュー反映後・推奨順）

1. **進捗の正本 `progress.json`（費用も内包）＋ `progress.events.jsonl`（追記）＋ `progress.md`（再生成）＋ `article:status`**（課題1・4・7 をまとめて回収。各工程は append-only でイベントを残し、JSON/MD は再生成。サブエージェントに JSON を直接書かせない）
2. **completion-report.md の保存**（課題7。`runs/<id>/` に閉じる。テンプレ化）
3. **factcheck 前の方向性ゲート direction-check.md**（課題3。create 直後=`draft.md` / refine 後=`final.md` を読む）
4. **factcheck 差分スキップ用の snapshot / changed sections / claim hash**（課題2・3。`claims.json` 不在でも `claims.raw.json`＋snapshot で判定できる経路を確保）
5. **コマンド承認の実運用監査**（課題6。allowlist に合わない叩き方＝`cd &&`/`npx`/相対パスを潰す。設定変更は最小限）
6. **見積もり・Claude Code 使用量は実績/手動ベース**（課題3-B・5。factcheck/build は API ログに出ないため `progress.json` 実績が溜まってから。Claude Code 使用量は手動・参考値）

## codex レビュー（80→88点）で直した肝

1. **進捗ログの正本と書き込み経路** … 正本は `progress.json` / `meta.json.steps`（`progress.md` は再生成）。さらに**サブエージェントに JSON を直接書かせず**、`progress.events.jsonl` 追記 or `progress:event` CLI ヘルパ経由に一本化（競合・破損回避）。
2. **7工程を誰が記録するか** … `WorkflowEvent` は create/evaluate/revise 系のみ。残り工程は CLI/サブエージェント共通のイベント記録で覆う。
3. **`claims.json` 不在でも差分判定できる** … 課題の実例（normalize 前の再 factcheck）に合わせ、初回 normalize を運用必須にするか、`claims.raw.json`＋`factcheck.snapshot.md` での判定経路を用意。
4. **コストの正本は二重化しない** … `progress.json` に寄せ、`cost.json` を作るなら派生（再生成）扱い。
5. **公開台帳と完成報告を混ぜない** … 完成報告は `runs/<id>/completion-report.md` に閉じ、`export/index.json`（record-publication の公開台帳）には足さない。
