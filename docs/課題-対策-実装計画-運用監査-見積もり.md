## スコープ判定（先に「もう済んでいること」を明確化する）

新規コードを足す前に、対策案で挙げた手当ての**既存状態**を確認した結果、大半は実装済み:

- **allowlist は新コマンド含め整備済み**（[templates/.claude/settings.json](../templates/.claude/settings.json)）: create/refine/evaluate/revise/resume/review/import/update-diff/claims-normalize/claims-recheck/verify-artifacts/review-editorial ＋（優先度1〜4で追加）status/progress:event/completion-report/direction-check/factcheck-scope。

- **編集長の「コマンド実行の作法」**（[templates/.claude/agents/article-editor-in-chief.md](../templates/.claude/agents/article-editor-in-chief.md)）に既に記載: 1 Bash=1 コマンド・`cd &&`/`|`/`;`/`echo` で連結しない・`npx` を前置しない・`--help` を叩かない。

## 優先度5: コマンド承認の運用監査（ドキュメント）

### D5-1. howto に「承認の棚卸し（audit）」小節を追加
- 位置: 「承認回数を減らす」節の末尾。
- 内容（対策案 §6-A）:
  - **承認過多の主因は「コマンドを足りない」ではなく「allowlist にマッチしない叩き方」**である旨を明記。
  - allowlist に当たらない典型パターンを列挙して潰す指針: `cd ../foo && llm-task-router ...`（`cd &&` 連結）/ `npx llm-task-router ...` / 相対パス起動 / パイプ・複合コマンド。→ **カレントディレクトリ基準で1コマンドずつ直接実行**に統一。
  - Claude Code に **`/fewer-permission-prompts` スキルがあれば**それで棚卸しできる（実際に出たプロンプト履歴をスキャンして allowlist 追加候補を提示）。本リポジトリ同梱ではなく Claude Code 側の機能なので「あれば使う」と条件付きで書く。**追加は監査で「正当だが未許可」と判明したパターンに限る**（むやみに広げない）。

### D5-2. 「意図的にプロンプトを残す」一覧の抜けを埋める（監査で発見）
- howto の当該リストは現在 export / factcheck-stamp / build-verifier の3つ。**`article:record-publication`（公開台帳の更新＝公開相当）も非 allowlist でプロンプト維持**だが、この中心リストに載っていない（update-article 節には記載あり）。**howto のリストへ追記するだけ**にする。
- **編集長定義（article-editor-in-chief.md）には触らない**: record-publication は update-article スキル側のコマンドで、編集長のコマンド早見には元々載っていない（意図的）。ここに足すとスコープクリープになるため**しない**。検証で確認済み（編集長定義に record-publication の記載なし＝設計どおり）。

### D5-3.（コード変更なし）
- allowlist は現状で過不足なし（優先度1〜4 で追加済み・状態変更系は除外済み）。**設定は触らない**。

## 優先度6: 見積もり・Claude Code 使用量（ドキュメント＋運用）

### D6-1. 「所要・コストの見方」を howto に明記
- `article:status` の表で **per-step の所要(ms)・概算$・合計**が既に見えること、出どころは `progress.json`（CLI 工程は自動、factcheck/build-verify は編集長の `progress:event` 記録）であることを書く。
- **概算コストは `models.yaml` の価格表依存の概算**である旨（対策案 §4-B）。`article:status` 表にも既に「概算」と出る。

### D6-2. factcheck/build の所要見積もりは「実績ベースで後日」（対策案 §3-B）
- ⚠ factcheck/build-verify は**サブエージェントの作業時間が主**で `router.log`（API コール）には出ない。`router.log` 平均で見積もれるのは create/evaluate/revise 系のみ。
- 現状は **`progress.json` に各工程の実 elapsed が溜まる段階**。横断集計（「factcheck 目安 約N分」）は**データが溜まってから別タスク**にする。本計画では**作らない**（憶測の数字を出さない）。実装するなら「複数 run の `progress.json` を集計して step ごとの中央値を出す」だが、それは優先度6の将来拡張。

### D6-3. Claude Code 使用量は手動・参考値（対策案 §5）
- Claude Code（外側オペレーター/サブエージェント）のトークン使用量は `router.log` に**含まれない**（別系統・本リポジトリ外）。自動取得しない。
- 運用ルールを明記: 必要なら編集長が Claude Code の `/cost`（セッション情報）を**完成報告の editor 欄（総評）**に手で転記する、または工程の所要・概算規模（読み込み行数・実行コマンド数）を `progress:event --note` に1行残す。**完全な課金額は取得不能**な前提で「参考値」と明示。
